# 先验分析与合成流水线对齐说明

> 本文档解释 `tip_i2v_data_analysis`（先验分析）和下游 Benchmark 合成流水线之间的"对齐"关系——即上游挖出了什么、下游怎么用、为什么这么设计。

---

## 一、大白话讲：这俩东西到底在干嘛

### 1.1 打个比方

想象你要给一个餐厅出一份菜单（Benchmark 题目）。有两种做法：

- **拍脑袋做法**：厨师觉得什么好就写什么，想到"满汉全席"就上，跟顾客实际爱吃什么完全无关。
- **数据驱动做法**：先去真实顾客的点单记录里挖一遍——顾客最爱点什么菜、口味偏甜还是辣、一桌平均几个人、是否在意摆盘——然后基于这些数据来设计菜单。

我们项目选的是第二种。

| 角色 | 对应组件 | 干的事 |
|------|---------|--------|
| **去餐厅调研顾客点单** | `tip_i2v_data_analysis`（先验分析） | 从 181 万条真实 I2V 用户请求中统计分布 |
| **基于调研设计菜单** | Benchmark 合成流水线（Stage 0–6） | 拿着统计结果来合成 1200 道评测题 |
| **调研报告** | `prior_package.json` | 两者之间的数据契约：一份结构化 JSON |

### 1.2 一句话总结

> **先验分析产出一个 `prior_package.json`，合成流水线的每一步都必须从这个文件里取数据做决策，而不是凭空编造。这就是"对齐"。**

## 二、先验分析产出了什么（上游输出）

先验分析流水线 5 步走完后，最终交付物是一个 JSON 文件，核心内容如下：

```
prior_package.json
├── global_distributions          ← 全局词汇分布（名词/动词/形容词各用了多少次）
├── global_visual_prior           ← 全局图像特征（主体数量、场景类型、镜头角度...）
├── pika_camera_distribution      ← 真实用户用了哪些运镜命令（zoom in 最多）
├── pika_motion_distribution      ← 真实用户设的运动强度（多数是 1-2）
├── dimension_priors[]            ← 每个维度的专属先验：
│   ├── concept_distributions     ←   常见主体/属性类型/动作词/方向词
│   ├── visual_prior              ←   适合这个维度的图像构图特征
│   ├── structural_templates      ←   真实 prompt 的句式模板
│   ├── seed_examples             ←   标注好的高质量真实样例（few-shot 用）
│   └── constraints               ←   维度约束规则（最少几个主体、合法变化类型...）
└── dimension_cooccurrence        ← 维度共现率（一个 prompt 同时涉及两个维度的比例）
```

**大白话：** 这玩意就是一份"用户画像报告"，只不过画像对象不是人，而是"I2V 用户群体写 prompt 的行为模式"。

## 三、合成流水线怎么消费这些数据（下游输入）

合成流水线有 7 个阶段（Stage 0–6），几乎每一步都要查 `prior_package.json`。下表逐步说明：

### Stage 0：词表库构建

| 用到的先验字段 | 怎么用 | 大白话 |
|---|---|---|
| `concept_distributions.target_subject` | 高频主体作为核心词表的"锚点" | 真实用户最爱画的东西（人、猫、狗、车），词表库必须优先收进去 |
| `global_visual_prior.typical_subject_categories` | 确定物体类别的权重 | 按真实热度排序，不能让冷门概念反客为主 |
| `pika_camera_distribution` | 运镜词表的组成和权重 | zoom in 用的多就多出 zoom in 的题，不能随便编 |

**如果不对齐会怎样？** 词表库收了一堆"航空母舰""量子计算机"这种真实用户从来不输入的概念，出的题跟真实场景脱节。

### Stage 1：题目规划（配额分配 + 概念采样）

| 用到的先验字段 | 怎么用 | 大白话 |
|---|---|---|
| `concept_distributions.*` | 80/20 常见/罕见比例的来源——频率 top70% = 常见 | 大部分题用用户常写的概念，小部分用罕见但合理的 |
| `visual_prior.scene_type_distribution` | 场景采样权重 | 真实用户喜欢"白天公园"就多出公园题，不是平均分配 |
| `visual_prior.shot_type_distribution` | 首帧构图的分布 | 真实用户的图大部分是中景 → 首帧设计也以中景为主 |
| `structural_templates` | LLM 生成 I2V prompt 时参考的句式 | 真实用户习惯写"a dog walks to the left"，合成的 prompt 也应该长这样，不要搞成学术论文体 |
| `seed_examples` | LLM few-shot 上下文 | 直接把真实标注好的样例塞给 LLM 当示范 |
| `constraints` | 维度内禁止规则 | 比如"运动绑定维度不允许出现局部动作词"——这是从真实数据里发现的边界 |

**如果不对齐会怎样？** LLM 随心所欲地编题，生成一堆现实中根本不存在的 I2V 场景，评测结果无法反映模型在真实使用中的表现。

### Stage 2–5：T2I 生成 → 图像质检 → I2V 定稿

| 用到的先验字段 | 怎么用 | 大白话 |
|---|---|---|
| `visual_prior.camera_angle_distribution` | 生成首帧图像时选镜头角度 | 大部分图用平视（真实数据最多的角度） |
| `visual_prior.background_separability_distribution` | 图像质检时检查前景-背景可分离度 | 先验告诉我们真实图像的背景分离度分布，太糊的不合格 |
| `dimension_cooccurrence` | 确认单道题只测一个维度、不混 | 真实 prompt 里"动作+运镜"经常一起出现（共现率 8%），设计时要刻意隔离 |

## 四、核心对齐原则

### 原则 1：先验是唯一真值来源

> 合成流水线中任何"比例"、"权重"、"选择"都必须能追溯到 `prior_package.json` 的某个字段。不允许出现"我觉得 zoom in 应该出 50%"这种拍脑袋数字。

### 原则 2：外部资源只做扩展，不做主力

流水线还用了 WordNet（物体词库）、Visual Genome（空间关系库）、COCO 80 类等外部资源，但这些的角色是：

| 角色 | 说明 |
|---|---|
| **主力** | 先验包中的真实分布 → 决定"出什么题"和"出多少" |
| **扩展** | WordNet/COCO → 补充先验包覆盖不到的长尾概念 |
| **校验** | ConceptNet → 检查概念组合是否合理（狗能跑但不能飞） |

**大白话：** 先验包定方向，外部资源查漏补缺。如果反过来让 WordNet 定方向，题目就会变成"词典里有什么就考什么"，跟用户实际用法脱节。

### 原则 3：句式必须还原真实用户的表达习惯

合成流水线生成的 I2V prompt 应该长得像真人写的，而不是机器模板。对齐方式：

- `structural_templates` 提供真实句式（如 `"a {noun} {verb} to the {direction}"`）
- `seed_examples` 提供完整样例做 few-shot
- 生成后可以跟真实分布做 KL 散度校验

### 原则 4：难度分桶必须反映真实分布

比如先验告诉我们：
- 52% 的真实 prompt 只涉及 1 个主体
- 26% 涉及 2 个主体
- 极少数涉及 3+ 主体

那么合成流水线的难度桶设计也必须：简单桶(1主体)比例最大，困难桶(3主体)比例最小。

## 六、举个完整例子

以 **Attribute Binding（属性绑定）** 维度为例，走一遍对齐流程：

### 先验分析发现了什么

```
dimension_priors["attribute_binding"]:
  - concept_distributions.target_subject: person(40%), dog(15%), cat(10%), car(8%)...
  - concept_distributions.attribute_type: color(35%), state(25%), wearing(20%)...
  - visual_prior.subject_count: 1个(55%), 2个(30%), 3个(10%)...
  - structural_templates: "a {noun} with {adj} {noun} {verb} to {adj}" (9.6%)
  - seed_examples: "a woman with blonde hair turns brunette in a park" ...
  - constraints: min_subjects=1, valid_types=["color","texture","state","wearing","size"]
```

### 合成流水线怎么用

1. **Stage 0（词表）**：物体词表中 person/dog/cat/car 被标记为 common（来自先验），corgi/tabby 被标记为 uncommon（来自 WordNet 扩展）
2. **Stage 1（规划）**：
   - 200 题配额中，80% 用 common 概念（person + color 变化），20% 用 uncommon（corgi + texture 变化）
   - 首帧规划中 55% 用单主体方案、30% 用双主体方案（对齐 subject_count 分布）
   - LLM 收到 seed_examples 做 few-shot，句式参考 structural_templates
3. **Stage 2（T2I）**：生成首帧时，构图中景为主（对齐 shot_type 分布）
4. **Stage 4（质检）**：VLM 检查首帧是否满足 constraints（至少 1 个主体、属性可见）
5. **Stage 5（定稿）**：I2V prompt 最终形式 = 真实用户句式风格，长度 8-25 词

### 如果不对齐（反面教材）

- ❌ 物体全从 WordNet 随机抽 → 出一堆"孔雀鱼颜色变化""针鼹质地改变"，真实用户根本不这么用
- ❌ 难度桶平均分配 → 单主体/双主体/三主体各 33%，跟真实分布（55/30/10）严重偏离
- ❌ prompt 风格像学术摘要 → "The chromatic attribute of the primary subject undergoes a transition..." 而不是 "the dog turns black"

## 七、常见误解澄清

| 误解 | 实际情况 |
|------|---------|
| "先验分析就是做数据清洗" | 不是。它做的是**语义级分析**（用 VLM+LLM 解析每条样本的维度可评测性），远比清洗复杂 |
| "合成流水线可以不用先验包、直接让 GPT 编题" | 技术上可以，但出来的题分布不合理、句式不自然、难以对标真实场景 |
| "prior_package 是一次性的" | 是的——分析一次，下游反复使用。但如果分析样本量扩大（1000→10000），需要重跑 |
| "对齐 = 硬编码" | 不是硬编码。是用先验数据作为**采样权重**和**约束规则**，LLM 仍有创造空间 |
| "WordNet/COCO 不重要" | 重要，但它们的角色是"扩展长尾"，不是"定义主流"。主流由先验包定义 |

## 九、总结

用最简单的话说：

1. **先验分析** = 去调研真实用户在"图生视频"这件事上到底输入什么内容、什么风格、什么习惯
2. **合成流水线** = 基于调研结果来批量制造考试题
3. **对齐** = 确保考试题的分布、风格、难度，都忠实反映真实用户的使用模式，而不是出题人自己编的

这样做的好处：用这套 Benchmark 评测出来的模型得分，才能真正代表模型在真实场景下的表现。如果题目跟真实用法脱节，分数再高也没意义。
