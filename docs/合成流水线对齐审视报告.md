# 合成流水线对齐审视报告

> 基于 [`项目介绍.md`](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/docs/项目介绍.md) 中已交付的 `prior_package.json` 字段，对照 [`Benchmark合成流水线详细方案.md`](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/docs/Benchmark合成流水线详细方案.md) 的实际设计，系统性审视对齐缺口与改进建议。

## 核心设计前提（已确认）

在阅读本报告之前，先明确以下已确认的评测设计决策：

1. **控制变量评测**：每道题严格只出现一个维度的变化，用该维度的专用评测器打分。不做跨维度组合题。
2. **固定模板 + 槽位填充**：提示词采用固定模板（名词/形容词/副词/动词的搭配顺序固定），LLM 不做自由创作，只做结构化 JSON → 自然语言翻译。
3. **80/20 常见/罕见的定义**：将 prior 词表按频率降序排列，取排名前 80% 的词条为 common，后 20% 的词条为 rare（按词条排名位置切分，不是按累计百分比切分）。
4. **词语搭配必须检查合理性**：槽位填充时需通过兼容性矩阵验证组合是否合理（如 `dog + fly` 不允许）。

## 一、合成流水线的不合理之处 / 先验分析未覆盖之处

### 问题 1：流水线偏离了"数据驱动"的核心定位（最大问题）

**现状**：合成方案的 Stage 0 把 WordNet 28787 物体、COCO 80 类、Places365、ConceptNet、Visual Genome 都列为主力来源，而 `prior_package` 中的 `concept_distributions.target_subject`、`global_visual_prior.typical_subject_categories`、`structural_templates`、`seed_examples` 只是辅助权重。

**冲突**：这与先验分析子工程"作为下游唯一数据驱动来源"的定位直接冲突。如果外部词表反客为主，先验分析的 ROI 被严重稀释。

**锚点**：`prior_package.concept_distributions` → Stage 0 词表构建

---

### 问题 2：多图模式 (multi-image) 在先验分析中完全空白

**现状**：合成方案里多图占比极高（Spatial 80%、Attribute/Action/Background 40%），但 TIP-I2V 整个数据集是 **单首帧 + 单 prompt** 的结构。

**缺口**：先验分析的 6 个 [`DimensionPrior`](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/src/utils/schema.py) 没有任何字段刻画多图组合。约 40% 的题目是"凭空"设计的，没有真实分布背书。

**锚点**：`prior_package.dimension_priors[]` → 无 multi-image 相关字段

---

#### 问题 3：维度共现矩阵的用途需重新定位

**现状**：`prior_package.dimension_cooccurrence` 是 6×6 的真实组合频率。

**已确认设计**：每题只测一个维度（控制变量），不做跨维度组合题。

**正确用途**：共现矩阵用于**维度隔离**——共现率高的维度对（如 action + camera = 8%），出题时必须刻意压制另一维度的变化（如 Action 题强制 `camera remains fixed`），确保评测纯度。

**当前缺口**：合成方案没有显式引用共现矩阵来制定维度隔离规则。

**锚点**：`prior_package.dimension_cooccurrence` → Stage 1 维度隔离规则

---

#### ~~问题 4：structural_templates 与 seed_examples 入口缺失~~（已被设计决策化解）

**原问题**：流水线 Stage 1 Step 3 让 LLM 自由生成 prompt。

**已化解**：采用固定模板 + 槽位填充方案后，模板本身直接来自 `structural_templates` 高频 top-N，LLM 仅做翻译而非创作。seed_examples 可用于验证模板覆盖率。此问题不再存在。

**锚点**：`prior_package.dimension_priors[].structural_templates` → 直接作为题目模板来源

---

### 问题 5：兼容性矩阵的来源选错了

**现状**：Stage 0 用 ConceptNet（CapableOf / AtLocation）+ LLM 校验来构建兼容性矩阵。

**缺口**：TIP-I2V 中主体-动作、主体-场景、属性-主体的真实共现本身就是一份天然兼容性矩阵——而先验分析没有提取这种"跨槽位共现"。

**建议**：先验工程应新增 **Conditional Co-occurrence Priors**（Part E）。

**锚点**：`prior_package` 无此字段 → Stage 0 `compatibility_matrix.json`

---

#### 问题 6：Common/Uncommon 罕见度标签未在先验中预计算

**已确认定义**：将词表按频率降序排列，取排名前 80% 的词条 = common，后 20% = rare。这是按词条排名位置切分，不是按累计百分比。

**缺口**：`prior_package` 中没有 `frequency_tier` 字段，[`ConceptDistribution`](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/src/utils/schema.py) 只有 count 和 pct。下游拿到数据后需自己排序并切第 80% 分位。

**建议**：先验工程在 Step 5 汇总时，对每个 concept_distributions 列表按 count 降序排列后，自动在第 80% 分位处切分，给每个条目打上 `frequency_tier: common/rare` 标签，下游开箱即用。

**锚点**：`prior_package.dimension_priors[].concept_distributions[].entries` → 补 `frequency_tier`

---

### 问题 7：可评测性约束没有透传

**现状**：[step4_joint_analysis.py](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/src/step4_joint_analysis.py) 的 `DIM_CONSTRAINTS` 已定义"双约束可评测性"。

**缺口**：合成方案 Stage 1 采样时没有强制要求所选概念组合在 prior 中被判定为 evaluable。可能合成出"先验中标注为不可评测"的题目（典型：单主体场景出 spatial_relation 题）。

**锚点**：`prior_package.dimension_priors[].constraints` → Stage 1 采样逻辑

---

### 问题 8：风格 / 画风分布未提取

**现状**：TIP-I2V 来自 Pika Discord，prompt 中包含大量 `cinematic / anime / 3D render / photorealistic` 等风格修饰。

**缺口**：先验分析没有提取风格分布。合成方案 Stage 2 一刀切写 `photorealistic`，与真实用户偏好脱节。

**锚点**：`prior_package` 无 `style_distribution` → Stage 2 T2I prompt 风格

---

### 问题 9：多主体的"类别组合"分布缺失

**现状**：S2A1、S2D2 等"双主体"题型大量存在。

**缺口**：`prior_package` 只统计了 `target_subject` 的边际分布，没有 (subject_a, subject_b) 的类别共现（dog+cat、man+woman 等）。流水线只能从外部猜测。

**锚点**：`prior_package.dimension_priors[].concept_distributions.target_subject` → 缺 pair 分布

---

### 问题 10：背景迁移场景对 (scene_pairs) 缺先验

**现状**：Stage 0 列出 `scene_pairs.json`（伦敦↔巴黎、海边↔山）。

**缺口**：`BackgroundChangeSlot` 只统计了 `to_state` 边际分布，没统计 (from_state, to_state) 配对。Background Type B 题目的语义合理性无据可依。

**锚点**：`prior_package.dimension_priors["background_dynamics"].concept_distributions` → 缺 pair

---

### 问题 11：pika_motion_distribution 只服务于 View，但其本质是 Motion 强度先验

**现状**：Pika 的 `-motion 1/2/3/4` 实际是用户对主体运动幅度的偏好。

**缺口**：当前流水线只把它用在 View Transformation。Motion Binding 的难度梯度（subtle / moderate / strong）应该由 pika_motion 派生，目前未利用。

**锚点**：`prior_package.pika_motion_distribution` → Stage 1 Motion Binding 难度设计

## 二、关于 WordNet / COCO 的取舍建议

### 总体判断

**WordNet/COCO 不应作为主力来源，但可作为"长尾扩展工具"。**

| 角度 | 偏向 | 理由 |
|------|------|------|
| 真实分布忠实度 | 反对主力引入 | 脱离 TIP-I2V 用户分布 |
| 长尾覆盖 | 支持适度引入 | TIP-I2V 10K Eval 切片对低频概念覆盖不足 |
| T2I 可生成性 | COCO 有工具价值 | COCO 类别是"T2I 易生成白名单" |
| 评测稳健性 | 支持适度引入 | WordNet 上下位词可制造可控难度梯度 |

### 推荐折中方案

1. **Common 桶（80%）严格使用 prior_package**：物体、属性、动作、场景全部从 `concept_distributions` 高频项采样，加权 = pct。
2. **Uncommon 桶（20%）使用 WordNet 的"语义近邻扩展"**：例如先验中高频是 `dog`，则 uncommon 用 `corgi / husky / dachshund`（同 hypernym 子树的下位词），而不是引入 prior 中完全不存在的词类。既制造长尾，又保持语义连续性。
3. **COCO 仅作 T2I 可生成性过滤器（白名单/黑名单）**，不作为采样源。
4. **ConceptNet 不必要**：兼容性矩阵从 TIP-I2V 真实共现派生，再用 LLM 补缺，比 ConceptNet 通用知识更贴近 I2V 场景。
5. **Places365 不必要**：用 prior 的 `scene_type_distribution`（lighting × weather × time）+ 高频名词中的场景词即可。
6. **Visual Genome 仅在 Spatial Composition 维度备用**，且只用 spatial relations 子集（10 个 predicate）。

> **一句话：TIP-I2V 是 prompt 分布的真值，WordNet 只是一个"罕见度调节器"。**

## 三、参考 T2V-CompBench / T2I-CompBench 的题目构建建议

### 建议 1：固定模板 + 槽位填充（已确认采纳）

**来源**：T2I-CompBench++ 范式  
**当前状态**：已确认为核心设计决策

具体做法：
- 每个维度定义 5–10 个固定模板（直接来自 `structural_templates` 高频 top-N）
- 槽位值从 prior 词表按频率采样，填充前通过兼容性矩阵校验
- LLM 仅做 JSON → 自然语言翻译，温度=0，无创作自由
- 变化类型限制为几种固定方式（如前/后/左/右移动、颜色切换、状态开关等）

**示例**（Attribute Binding S2A1 模板）：
```
"The {color1} {subject} turns {color2} while the {color3} {subject} remains unchanged."
```
槽位从 prior `concept_distributions.target_subject` 与 `attribute_type=color` 填充。

---

### 建议 2：80/20 common/rare 按词条排名切分（已确认）

**来源**：T2V-CompBench 经验  
**已确认定义**：词表按频率降序排列，前 80% 词条 = common，后 20% 词条 = rare

- 配额硬约束：每维度 200 题 = 160 common + 40 rare
- rare 中 30 题由 prior 长尾贡献，10 题由 WordNet 下位词扩展（同 hypernym 子树的语义近邻）
- 先验工程应直接产出带 `frequency_tier` 标签的词表，下游开箱即用

---

### 建议 3：对比组系统化为"pair_id 强约束"

**来源**：T2V-CompBench 经验  
**适配改造**：`contrastive_pair_id` 必填，`contrastive_role ∈ {A, B}` 且 A/B 必须同时存在

- 200 题 = 100 对，便于评测时做"模型偏向差分"
- 配对维度优先级：方向（left↔right）、状态（white→black ↔ black→white）、运镜（zoom in ↔ zoom out）

---

### 建议 4：k-tuple 难度可控

**来源**：ConceptMix 经验  
**适配改造**：将当前的 S×A、S×D 桶模型显式化为公式

```
难度 = 主体数 × 同时绑定的概念数 × 推理复杂度
```

每个难度档明确配额，避免不均衡分配。

---

### 建议 5：程序化合成 + LLM 仅做翻译（已确认）

**来源**：Generate Any Scene 经验  
**当前状态**：已确认为核心设计决策，与建议 1 一脉相承

- **第一层**（程序化）：从 prior 词表 + 兼容性矩阵 + 模板，确定性地生成 Question Plan（结构化 JSON）
- **第二层**（LLM）：仅把 Question Plan 翻译为自然语言 prompt，温度=0，无创作自由

好处：可重放、bug 可定位、概念覆盖率可统计。

---

### 建议 6：图像质检引入 BLIP-VQA 风格问句

**来源**：T2V-CompBench 经验  
**适配改造**：从 Question Plan 自动生成是非题

- 示例：`"Is there a white dog on the left? (Yes/No)"`
- VLM 回答 Yes/No，全 Yes 才通过
- 与下游评测对齐，验证标准统一

---

### 建议 7：每维度独立的"配额 × 难度 × 罕见度"三维矩阵

**来源**：T2V-CompBench 表格化设计  
**适配改造**：在每个维度方案中明确表格

```
                  common(80%)              rare(20%)
Easy(40%)         64 题                    16 题
Medium(40%)       64 题                    16 题
Hard(20%)         32 题                    8 题
```

避免"难度桶详尽但配额留白"的现状。

---

### 建议 8：评测题目的"先验自洽性检查"

每道生成题目在落库前自动校验：
- 所采样的概念组合在 prior `dimension_priors[dim].evaluable_samples` 中是否真实出现过
- 若未出现且非有意 rare，则丢弃或标记 `out_of_distribution`

这是合成流水线对先验分析的"反向使用"，目前完全缺失。

## 四、给先验分析子工程的补强清单

以下字段建议落地到 [step4_joint_analysis.py](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/src/step4_joint_analysis.py) 和 [src/utils/schema.py](file:///d:/projects/I2V-CompBench/tip_i2v_data_analysis/src/utils/schema.py)：

| 编号 | 待补字段 | 用途 | 下游 Stage |
|------|---------|------|-----------|
| P1 | `frequency_tier ∈ {common, rare}` 按词条排名前80%/后20%切分 | 下游开箱即用的常见/罕见标签 | Stage 1 |
| P2 | `subject_pair_distribution` 双主体类别共现 | S2 系列题型采样 | Stage 1 |
| P3 | `subject_action_cooccurrence`、`subject_scene_cooccurrence`、`subject_attribute_cooccurrence` | 替代 ConceptNet 兼容性矩阵 | Stage 0 |
| P4 | `style_distribution`（photo / anime / 3D / cinematic） | T2I prompt 风格选择 | Stage 2 |
| P5 | `change_pattern_pairs`（from→to 配对频率） | Attribute/Background 题型 | Stage 1 |
| P6 | `pika_motion_intensity` 与 motion_binding 难度的映射 | Motion 强度梯度 | Stage 1 |
| P7 | `evaluable_combinations`（每维度真实出现的合法组合白名单） | 自洽性检查 | Stage 1/4 |
| P8 | `dimension_isolation_rules`（基于 dimension_cooccurrence 的维度隔离规则） | 控制变量时压制共现维度干扰 | Stage 1 |
