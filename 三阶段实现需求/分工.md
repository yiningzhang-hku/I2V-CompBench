# I2V-CompBench 协作毕业设计分工方案

## 一、分工边界

以数据集为交接点，两人贡献边界清晰可区分：

| 阶段 | 产物 | 归属 |
|------|------|------|
| Phase 1（先验数据准备）| prior_package.json、reference_bank/、candidate_recipes.jsonl | 数据集构建方 |
| Phase 2（数据集合成）| question_plans.jsonl、first_frames/、final_prompts.jsonl、v2_t2vlike/ | 数据集构建方 |
| ——— 交接点 ——— | **phase3_manifest.jsonl + BenchmarkSample 结构化样本** | ——— |
| Phase 3（模型评测）| per_sample_scores.jsonl、dimension_scores.json、human_correlation_report.json | 评测体系方 |

**核心原则**：数据集构建方交付数据集，评测体系方消费数据集。数据集本身是天然交接边界。

---

## 二、各自闭环故事

### 数据集构建方（Phase 1 + Phase 2）

**核心叙事**：如何系统性地构建一个可控、可评测的 I2V 组合性 benchmark 数据集。

```
Phase 1（先验提取）
  → Phase 2（模板化合成 + VQA质检 + Prompt定稿）
    → 独立验证（人工抽检 + 分布分析）
    → 交叉验证（借用Phase 3评测，证明维度/难度区分性）
```

Phase 3 在此方论文中定位为**"数据集有效性验证"**，非"评测系统搭建"。用小规模实验证明：
- 数据集的维度标签能区分模型不同能力
- 数据集的元数据（预期失败模式）与实际模型失败行为吻合
- 数据集的难度分层与模型表现下降趋势一致

### 评测体系方（Phase 3）

**核心叙事**：如何基于结构化 benchmark 数据集构建 I2V 组合性评测体系，并通过多模型实验揭示当前 I2V 模型的能力边界。

```
理解数据集元数据（Phase 1+2 背景）
  → Phase 3（评测指标设计 + 多模型部署 + 综合实验）
    → 独立验证（指标-人类一致性）
    → 交叉验证（用数据集生成视频，证明多模型可区分）
```

Phase 1+2 在此方论文中定位为**"理解并利用数据集元数据设计评测指标"**，非"重新构建数据集"。

---

## 三、交叉验证关系

双方互相作为对方的"下游消费者"验证各自产物价值：

| 方向 | 使用对方产物 | 证明的命题 |
|------|------------|-----------|
| 数据集方 → 借用 Phase 3 | 调用评测系统跑分 | 数据集的诊断价值（维度区分性、难度区分性、失败模式可预测性） |
| 评测方 → 借用 Phase 2 | 用首帧图像+提示词生成视频 | 评测系统的有效性（指标-人类一致性、指标敏感性、跨模型可比性） |

**各自验证的核心命题不同，不可混淆**：
- 数据集方证明的是"数据集能区分模型能力"，不是"评测系统好用"
- 评测方证明的是"评测指标可靠有效"，不是"数据集好"

---

## 四、独立验证路径

不能完全依赖对方的系统做验证，需各有独立路径：

### 数据集方的独立验证

1. **人工抽检**：随机抽 50-100 条样本，人工判断 prompt 是否正确、图像是否包含目标主体、维度标签是否准确
2. **分布分析**：主体分布、动作类型分布、背景类型分布的多样性统计
3. **对比实验**：用同一模型在 benchmark 和随机 prompt 上分别生成视频，对比组合性错误率

### 评测方的独立验证

1. **指标-人类一致性**：自动评分与人工评分的相关系数
2. **指标敏感性**：好视频与差视频能被指标区分
3. **合成测试**：用构造的 trivial 测试用例验证指标边界行为

**两条独立验证路径 + 一条交叉验证路径 = 各自闭环完整，且互不依赖。**

## 五、共享概念处理

以下核心概念为双方共同设计，需各自从不同角度论述：

| 共享概念 | 数据集构建方角度 | 评测体系方角度 |
|---------|----------------|--------------|
| **Preserve-Transform 双轴** | 数据集设计原则：每条样本拆解为 Transform 变化约束 + Preserve 保持约束 | 评测设计原则：对 Transform 评 Execution 分，对 Preserve 评 Preservation 分 |
| **E/P/C 三层** | 定义"评什么"：样本元数据标注 expected_change、preserve_constraints、coherence 要求 | 实现"怎么评"：用 Grounding DINO 算 E 分、DINOv2 特征相似度算 P 分、光流连续性算 C 分 |
| **7 维度** | 设计维度并生成样本：语义槽位、prompt 模板、维度隔离规则 | 为每个维度设计评测指标：Attribute 用 VLM VQA，Motion 用 RAFT 光流，View 用全局运动分析 |
| **维度隔离** | 在 prompt 层面强制：forbidden_words、camera_blacklist、约束检查 | 在评测层面验证：检查非目标维度是否意外变化 |

**学术诚信原则**：各自论文中明确标注共享设计来源。例如"评测体系由本文与合作者共同设计，本文负责 benchmark 结构与样本格式设计，评测指标实现详见合作者工作"。

## 七、答辩准备

最可能被问的问题：**"你们两个的论文有什么区别？是不是同一个项目拆成两半？"**

建议各自准备一句话回答：

- **数据集构建方**："我的贡献是 benchmark 的构建方法——如何从真实用户数据先验出发，通过模板化合成、维度隔离和质量控制，生成可控、可评测的数据集。评测指标实现和多模型实验是合作者的工作。"

- **评测体系方**："我的贡献是评测体系的设计与实现——如何基于结构化数据集的元数据，构建 E/P/C 三层评分体系，并通过多模型实验揭示 I2V 模型的组合性能力边界。数据集构建是合作者的工作。"

---

## 八、注意事项

1. **谨慎使用"高质量"措辞**：数据集存在已知局限（生僻词、图像清晰度、尺寸不统一等），建议用"可控性"和"可评测性"替代"高质量"，局限在"局限性与未来工作"中坦诚说明。

2. **视频生成为共享基础设施**：双方均需部署 I2V 模型生成视频，这是基础设施工作而非贡献差异点。差异在于生成视频后做什么——一方验证数据集诊断价值，一方验证评测指标有效性。

3. **避免循环论证**：数据集有效性不完全依赖对方评测系统，评测有效性不完全依赖对方数据集。各自保留独立验证路径。
