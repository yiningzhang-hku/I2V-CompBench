## 二、Phase 1：先验数据准备

Phase 1 的任务用一句话说：拿到 Step 1–5 留下的结构化解析结果，不再调任何模型 API，纯靠规则把它们加工成 Phase 2 能直接抽题的配方文件。

代码在 `tip_i2v_data_analysis/` 下。前置步骤 `src/step*.py` 完成基础解析，Phase 1 增强步骤 `src/phase1/` 六个脚本接力完成剩余加工。

运行：`python main.py --step phase1 --config configs/config.yaml`（一键依次跑完全部 6 步），也可以单步跑 `--step patch` / `--step align` 等。

---

### Step 1 · 清单构建 `step1_manifest.py`

扫描 HuggingFace 上的 TIP-I2V 数据集，下载首帧图像到本地，同时把 prompt 里夹带的 Pika 平台参数（`-camera pan`、`-motion 3` 之类）剥离出来，只留纯自然语言部分。图坏了或 prompt 为空的样本打上 bad 标记隔离掉。

产出：
- `manifest_clean.jsonl`：每行是一条通过清洗的样本，带 sample_id（全局主键 `tip_XXXXXX`）、clean_prompt、pika_camera/pika_motion（只用来做运镜统计）、image_path。
- `images/` 目录：首帧图像。
- `bad_samples.jsonl`：不合格的。

sample_id 是后面所有步骤关联数据的唯一键。clean_prompt 去掉了平台参数，确保后续 LLM 只看到用户真正想表达的内容。

---

### Step 2 · 图像结构化解析 `step2_image_analysis.py`

用 VLM 逐张分析首帧：画面里有几个主体、分别在什么位置（9 宫格描述如 "upper-left"）、什么属性（颜色/材质/穿着）、是不是活体、主体之间什么空间关系、背景什么情况、镜头基线怎样。

产出 `image_parse.jsonl`，每行一个样本，核心字段：
- `subjects[]`：主体列表（name / position_in_frame / attributes / is_animate）
- `subject_relations[]`：空间关系
- `background`：前后景分离度、光照、天气
- `camera_baseline`：镜头类型、是否有刚性参照物

Phase 1 里 patch 从 position_in_frame 反推 bbox；refbank 根据 subjects 裁剪素材；align 用 name/description 做匹配。

---

### Step 3 · 文本意图解析 `step3_text_analysis.py`

用 LLM 分析每条 clean_prompt：判断涉及哪几类变化，抽取对应的结构化槽位。

产出 `text_parse.jsonl`，核心内容：
- 六个布尔标记：`involves_attribute_change` / `involves_action` / `involves_directed_motion` / `involves_spatial_relation_change` / `involves_background_change` / `involves_camera_movement`
- 六种槽位列表：attribute_change_slots / action_slots / motion_slots / spatial_relation_slots / background_change_slots / camera_movement_slots

Phase 1 靠这六个布尔做维度路由，靠槽位里的 target_subject 做图文对齐。

---

### Step 4 · 联合分析 `step4_joint_analysis.py`

交叉验证图像和文本：文本说有球在动，图像里到底有没有球？图像里有两个人，文本有没有描述他们的互动？产出六维可评估性判断和全局先验统计。

产出：`joint_analysis.jsonl` + `dimension_priors.jsonl` + 多个先验 JSON。Phase 1 的 priors_enhance 在此基础上扩展到第 7 维。

---

### Step 5 · 先验包组装 `step5_pool_and_report.py`

把 Step 4 散落的统计汇总成一份 `prior_package.json`——全局概念分布、维度先验、种子样例。Phase 1 的 priors_enhance 以此为基线，在上面加频率分桶和兼容矩阵。

---

### Phase 1 增强步骤（6 步）

以下六步接手 Step 1–5 的产物往上叠加。核心约束：**不调 VLM / LLM**，纯规则推导 + 几何近似。

#### P1.1 字段补齐 `patch_existing_outputs.py`

Step 2/3 的解析结果缺 Phase 1 后续要用的几个东西：图像侧没有数值 bbox 和 instance_id，文本侧没有主维度标签和防泄漏列表。

图像侧处理（逐样本逐主体）：
- 给每个主体写入 instance_id（没有的话用主体原始 id 充当）
- 把 "upper-left" 这种位置描述通过 9 宫格映射表翻译成 `[x0, y0, x1, y1]` 归一化坐标
- segmentation_quality 一律标 "low"（P0 没真实分割）；tracking_feasibility 根据 bbox 面积 + is_animate 推断（面积 <2% → infeasible，<6% → hard，<20% 且活体 → medium，否则 easy）
- 推断整个样本的 reference_potential：判断哪类素材可以从这张首帧里抽取

文本侧处理（逐样本）：
- 维度路由：按 interaction > attribute > action > motion > spatial > background > view 的优先级，从 involves_* 字段取第一个命中的作为 primary_dimension
- 防泄漏：查 LEAKAGE_TABLE 填入 forbidden_dimension_leakage
- 交互推断：action_slots 中 ≥2 个不同 target_subject 且 spatial_relation_slots 非空 → 推断存在 interaction

产出：`image_parse_v2.jsonl` + `text_parse_v2.jsonl`。

**为什么这样设计**：

- **9 宫格映射 vs SAM/Grounding-DINO**：Phase 1 的硬约束是「零 API 调用、纯确定性、幂等可重跑」。9 宫格是最小可用的几何近似方案；引入分割模型会让 Phase 1 与该约束直接冲突，且每次重跑结果不可复现。`geometry_mode` 配置位预留为 `mock_9grid` / `sam`，未来切换不破坏下游 schema。
- **LEAKAGE_TABLE 集中硬编码**：维度隔离是评测一等约束。一旦分散到模板由不同作者各写各的，「严格度」会随风格漂移，论文「评测可精确归因」的论点就站不住脚。
- **主维度优先级 interaction > attribute > … > view**：interaction 涉及 ≥2 主体，是数据池里最稀缺的，必须最先抢占；view 最容易被 LLM 误判（任何含「镜头」「画面」描述的句子都会触发 involves_camera），放兜底位避免抢戏。
- **优先级取「列表第一个命中」而非加权评分**：可解释、可审计——任意一条 recipe 的归类原因都能被简单脚本复现，不依赖任何浮点数。

#### P1.2 图文实例对齐 `align_instances.py`

图像侧有一组主体（带 instance_id 的），文本侧有一组目标主体名称（从各种 slots 里提取的）。两边没对应关系。这一步做名称软匹配把它们关联上，顺便诊断每个维度的可评测性。

对齐逻辑：从文本的 attribute_change_slots / action_slots / motion_slots / spatial_relation_slots / interaction_slots 中收集所有 target_subject 名称，然后逐个和图像主体的 name / description 做匹配。完全相等 → confidence=1.0，子串匹配 → 0.7，都不匹配 → 0.0。

样本级的 overlap_ratio = 匹配成功的文本主体数 ÷ 文本主体总数。

七维诊断：对每个维度判断 text 是否请求（involves_* 为 true）× image 是否支持（如 action 需要 has_animate，spatial 需要多主体且可区分）× 几何质量，输出 tool_status（valid / low_confidence / tool_uncertain / invalid_input）。

产出 `aligned_instances.jsonl`：aligned_subjects[] / overlap_ratio / evaluator_feasibility[] / dimension_isolation[]。

**为什么这样设计**：

- **名称软匹配（1.0 / 0.7 / 0.0）而非 embedding**：零 API 约束让 embedding 不可用。子串匹配已能处理 80% 的大小写、单复数、定冠词差异（`a red ball` vs `red ball` vs `the ball` 全部命中），剩余极端样本由后续 QC 兜底。
- **贪心匹配而非匈牙利**：图像主体 ≤5、文本主体 ≤5，匈牙利的最优性收益微小却让「为什么这两个对上」难以解释；贪心 + 出现序保证人工排查时一眼能复现匹配链路。
- **诊断与对齐放在同一步**：`tool_status` 直接依赖 `alignment_confidence`，拆成两步会让两份字段在 schema 演化时出现「对不上」的窗口期。
- **七维诊断的 image_support 条件硬编码**：每个维度对图像基线的真实需求不同（action 要 has_animate，spatial 要多主体且可区分），写死在代码里比让上游 VLM 自己判断更稳。

#### P1.3 多图素材抽取 `reference_bank.py`

多图评测模式需要给模型看参考图（单独的主体裁剪图或场景背景图）。这些从首帧中裁出来。

根据 image_parse_v2 中每个样本的 reference_potential 决定抽什么：

| 类型 | 做法 | 维度 |
|------|------|------|
| subject | 按 bbox 裁剪（5%外扩） | attribute/action/motion/spatial/interaction |
| attribute | 同 subject | attribute |
| object | 只裁非活体 | spatial/interaction |
| scene_reference_original | 原图复用 | background/view |
| scene_reference_inpainted | P0 跳过 | — |

asset_id 命名：主体类 `{sample_id}_{instance_id}_{suffix}`（subj/attr/obj），场景类 `{sample_id}_scene_orig`。

每个资产带 provenance（source_sample_id / source_image_path / extraction_method / extraction_params / created_by / created_at）和 quality（resolution / aspect_ratio / quality_score / quality_flags）。

产出：`assets.jsonl` + `reference_bank/` 目录下的图像文件。

**为什么这样设计**：

- **bbox 外扩 5%**：crop 太紧会切掉主体轮廓导致边缘伪影，太松又带入背景诱导下游「看背景反推空间关系」的捷径。5% 是经验上「保留主体完整边缘且不显著带背景」的平衡。
- **quality_score = clamp(0.5 + min(bw, bh), 0, 1)**：用 `min(bw, bh)` 而不是面积或 `max`，是为了防「窄长条形假大」——画面横亘的细线 bbox 面积可能不小，但裁剪后短边过窄无法 resize 到 854 长边。强迫两边都要够大才与下游 PNG long_edge 约束对齐。
- **每个资产必带 provenance**：reviewer 抽检要复现 crop 时，必须能区分「9 宫格旧版」和「未来 SAM 版」——同一个 sample_id 在不同几何模式下会产出不同 bbox，没 `extraction_method` + `created_at` 就无法定位是哪一版规则。
- **scene_reference_inpainted 在 P0 跳过**：抹除主体的 inpainting 需要扩散模型，与「零 API」冲突；先空着字段位等 P1 再接入。

#### P1.4 先验增强 `priors_enhance.py`

Phase 2 采样时需要几类 Step 5 没有覆盖到的统计：

1. **frequency_tiers.json**：主体/动作/属性按频率分三档（head 前50% / torso 50–85% / long_tail 后15%）。Phase 2 用它控制 common/rare = 80/20 的配比。

2. **subject_pair_distribution.json**：哪些主体经常共现（top-200 对）。组合多图题时参考，不拼现实中没见过的组合。

3. **multi_reference_priors.json**：有多少样本能走多图路线（对齐主体≥2）、各类资产各有多少。

4. **compatibility_matrix.json**：7×7 维度兼容矩阵。compatible / leakage_risk / incompatible 三态。如 attribute × motion = incompatible。

产出在 `phase1/priors/` 目录下。

**为什么这样设计**：

- **三档累计频率 50% / 35% / 15% 而非 33/33/33**：head/torso/long_tail 对应 Zipf 长尾分布的天然分段。强行三等分会让 common 桶里塞进只出现一两次的概念，Phase 2 的 80/20 common/rare 配比就失去意义——分段点必须贴合真实曲线。
- **共现对取 top-200**：Phase 2 多图样本约 100–200 量级，top-200 已足够覆盖；扩到 top-1000 会让频次 ≤2 的低频对主导分布，反而引入采样噪声。
- **7×7 兼容矩阵硬编码而非数据驱动**：`attribute × motion = incompatible` 是论文论点（评测必须能精确归因），不允许数据反向影响。如果某天数据里这两类大量共现，结论应是「数据有问题」而不是「放宽矩阵」。
- **frequency_tiers 单独存为 JSON 而非塞进 prior_package**：Phase 2 build_quota 只关心分桶结果，不需要重读整份先验包；分文件让单独重跑某一步成本最小。

#### P1.5 候选配方生成 `recipes.py`

前面的步骤产出了对齐、资产、先验、隔离约束等散落文件。这一步把所有信息打包成 Phase 2 能直接采样的结构化 recipe。

生成条件：有 primary_dimension + 该维度 feasible=true + tool_status ≠ invalid_input。不满足的跳过。

核心决策（input_mode + source_type）：
- view / background → single_image + observed_single_image
- spatial / interaction 且有≥2主体资产且对齐≥2 → multi_image + derived_multi_reference
- attribute / action / motion 且有≥2主体资产 → multi_image + derived_single_image
- attribute / action / motion 且有1个资产 → single_image + derived_single_image
- 兜底 → single_image + observed_single_image

每条 recipe 还带 preserve_constraints（按维度预填保持约束）、contrastive_spec（4 类 baseline）、dimension_isolation、expected_difficulty（valid→medium，low_confidence→hard）。

产出：`candidate_recipes.jsonl`。**Phase 2 唯一的题目来源**。

**为什么这样设计**：

- **input_mode 自动决策而非 yaml 配置**：把「什么样的样本走什么模式」硬编码成规则，避免不同 reviewer 跑同一份 candidate 得到不同 benchmark；同时让「这条 recipe 为什么是 multi_image」可以逐条用决策树解释。
- **spatial/interaction 优先多图、view/background 强制单图**：前者天然涉及多主体，分图给参考态能减小语义歧义；后者是镜头/场景级变换，多图反而让模型困惑「以哪张为基准」。
- **expected_difficulty 由 tool_status 决定而非人工估难**：把「难度」绑到 evaluator 可执行性上，保证横向可比；人工估难会引入作者风格漂移。
- **PRESERVE_DEFAULTS 按维度预填**：保持约束是评测 P 分的判别清单，让 Phase 1 在生成 recipe 时就把「哪些方面不能变」固化进去，Phase 3 不必反推 prompt 才知道。
- **contrastive_spec 固定 4 类 baseline**：static_copy / random_motion / global_filter / camera_pan_cheat 覆盖论文 anti-shortcut 表的所有列；如果 baseline 集合各 recipe 不同，统计表就拼不齐。

#### P1.6 审计报告 `audit.py`

Phase 1 → Phase 2 的门禁检查。读取所有 Phase 1 产物，输出 7 节报告：

1. 样本规模（各文件行数）
2. 主维度路由分布（七维各多少）
3. 七维可评测性诊断（tool_status 四态分布）
4. 资产分布 + provenance 非空率（期望 100%）
5. Recipe 分布（按维度 / input_mode / source_type）
6. 隔离有效性（主维度不应出现在 forbidden 里，期望 0 违规）
7. 验收清单（pass/fail）

产出：`phase1_audit_report.md`。不达标就修再跑。

**为什么这样设计**：

- **7 节缺一不可**：每节对应一类风险——样本规模 → 完整性、路由分布 → 代表性、七维诊断 → 评测可达性、资产 provenance → 可审计性、recipe 分布 → 均衡性、隔离有效性 → 评测可信度、验收清单 → 出货门禁。漏掉任意一项都会让该类风险静默积累到 Phase 2/3 才暴露。
- **隔离违规 = 0 是硬门槛**：违规即说明 `LEAKAGE_TABLE` 推导链路存在 bug，往下跑会让所有 recipe 都带错；先修再放行比「打折出货」安全。
- **Markdown 而非 JSON**：审计报告主要受众是项目 owner 与论文作者，Markdown 在 CI 可读、git diff 友好；机器需要的关键统计已由 `priors/*` 与 `candidate_recipes` 字段层提供。
- **audit 不复用 patch/align/recipes 的代码**：审计是「独立见证人」，与生产代码共用函数会让同一处 bug 在两边都按错规则处理仍「自洽通过」。

## 三、Phase 2：Benchmark 数据集合成

> Phase 2 接力 Phase 1，把 `candidate_recipes.jsonl` 这一堆「可以出题的种子」一步步变成最终带图、带 prompt、带评测说明书的 `benchmark_dataset/`。流水线由 8 个模块串成单向管道，前一步的输出就是后一步唯一的输入。
>
> **代码位置**：`i2v_compbench/src/i2vcompbench/phase2/`
>
> **CLI 入口**：`python -m i2vcompbench.phase2.<module> --config configs/phase2.yaml` 或统一入口 `python -m i2vcompbench.main phase2 --mode pilot`。

### Step 1 · `build_quota.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `configs/phase2.yaml`：`mode`（pilot=20/dim、full=200/dim）、`input_mode_ratio`、`subtype_ratio`、`difficulty_ratio`、`rarity_ratio` |
| **处理逻辑** | 对每个维度按 `mode → subtype → difficulty → rarity` 四级展开，每一级用最大余数法 `_round_split` 把整数配额按比例切到子桶；motion 维度按 subtype 拆，其它维度按 input_mode 拆 |
| **产物** | `benchmark_dataset/quota_plan.json`：`QuotaPlan { mode, total_target, buckets: [QuotaBucket] }` |
| **关键字段** | `bucket_id = {dim}__{mode}__{difficulty}__{rarity}`、`target_count`、`contrastive_pair_required` |
| **下游消费者** | Step 2（按 bucket 抽 recipe）、Step 8（按 bucket 算缺口）、Phase 3 聚合分组键 |
| **工程用途** | 把抽象的「7 × 200 题」拆成几百个可执行的小桶，决定每个桶要采多少 recipe |
| **论文用途** | 「数据集分布表」直接来源；解释为何 spatial=80% 多图、view=95% 单图（数据驱动而非平均） |

**字段释义**：

- `bucket_id`：唯一定位一个配额桶的字符串主键。后续模块都以它为 join key。
- `target_count`：该桶需要采样出多少题，pilot 总和 = 140，full 总和 = 1400。比例不为 1 时会先归一化再分。
- `difficulty`：`easy / medium / hard`，按 40/40/20 切分。
- `rarity`：`common / rare`，按 80/20 切分，源自 Phase 1 `frequency_tiers.json`。
- `contrastive_pair_required`：该桶是否要求 A/B 成对采样（例如 attribute 维度的「red→blue」与「blue→red」）。

**为什么这样设计**：

- **最大余数法 vs 四舍五入**：保证 `target_count` 求和精确等于 `num_per_dimension × 7`，下游 audit 才能用减法直接得到缺口；多级换算下舍入会让总和漂移。
- **bucket_id 提前固化到 quota_plan**：汇总键在最上游定好，sample / export / audit 都用同一主键 join，不必每步重算配额。
- **七维度共用 difficulty/rarity、input_mode_ratio 维度独立**：难度与稀有度是横向可比的统一评价轴；输入模式分布与维度真实先验强耦合（view 95% 单图、spatial 80% 多图），共用一份比例会失真。
- **motion 维度退化为 subtype 分桶**：motion 在 TIP-I2V 真实分布里的分类轴是 `type_a/b/c`，不是 single/multi 二元；强套 input_mode 会把「绝对位移」与「相对位移」挤进同一桶。

### Step 2 · `sample_recipes.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `quota_plan.json` + Phase 1 的 `candidate_recipes.jsonl` + `assets.jsonl` |
| **处理逻辑** | ① 把 candidate_recipes 按 `(dim, mode/subtype, difficulty, rarity)` 建索引；② 用 `_BLOCKING_QUALITY_FLAGS` 把质量被否决的 recipe 直接排除；③ 多图 recipe 逐条校验 reference 资产的 `quality_score ≥ 0.4` 与 `is_clean_background`，至少 2 张达标；④ 按 `target_count` 在每个桶内抽样，contrastive 桶按 `source_sample_id` 分组成对消费 A/B；⑤ 缺口写入报告，分四类原因 |
| **产物** | `benchmark_dataset/sampled_recipes.jsonl` + `benchmark_dataset/quota_unfilled_report.json` |
| **关键字段** | `SampledRecipe { recipe_id, bucket_id, contrastive_pair_id, contrastive_role(A/B/A_only), reference_asset_ids[] }`；缺口报告 `{ bucket_id, target, sampled, gap, reasons[] }` |
| **下游消费者** | Step 3（生成 question plan）、Step 8（审计） |
| **工程用途** | 在配额约束下从真实 Phase 1 池子里抽题，绝不绕过 Phase 1 自己拼词表 |
| **论文用途** | 缺口报告作为「benchmark 仍可扩展方向」的证据；A/B 配对率支撑 contrastive consistency 实验 |

**字段释义**：

- `_BLOCKING_QUALITY_FLAGS = {low_alignment, missing_inpainted_scene, subject_not_visible, evaluator_infeasible}`：碰到任意一项即整条 recipe 出局。
- `_MULTI_IMAGE_SUBTYPES = {type_c_multi_motion}`：motion 维度也要走多图体检通道。
- `contrastive_role`：成对达成时分 `A` / `B`，落单则记 `A_only` 并在缺口报告里加 `contrastive_unpaired`。
- 缺口原因四类：`no_candidate`（桶内没有 recipe）/ `no_reference_asset`（多图但缺资产）/ `quality_below_threshold`（资产质量不达标）/ `blocking_flag`（候选都被 blocking 了）。**绝不静默降级 input_mode**。

**为什么这样设计**：

- **绝不允许 Phase 2 自拼词表**：benchmark 可信度建立在「题目源自真实 I2V 分布」——一旦启发式拼出新组合，reviewer 第一个质疑就是「怎么证明对应真实场景」。让 `candidate_recipes` 成为唯一题源，每条样本都能反查一条 TIP 视频。
- **多图 ≥2 张达标 + clean_background 阈值前置到抽样阶段**：在抽样阶段（而非 QC 阶段）剔除不可用 recipe，节省后续 T2I 与 VQA 调用。`construct_inputs` 才发现少图时已浪费一次 question_plan 构建。
- **A/B 原子化配对**：contrastive consistency 是 anti-shortcut 的关键证据，只采到 A 没采到 B 的样本在 contrastive 表里直接作废；「配对完整性」比「量多」更重要。
- **缺口分四类原因而非单个 gap 数字**：reviewer 关心「差在哪」而非「差几条」。四类分别对应四种 actionable 工单：补数据 / 补资产 / 调阈值 / 修 Phase 1 patch。

### Step 3 · `build_question_plan.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `sampled_recipes.jsonl` + Phase 1 的 `image_parse_v2.jsonl` / `text_parse_v2.jsonl` / `aligned_instances.jsonl` + `configs/templates/{dimension}.yaml` |
| **处理逻辑** | ① 按 `image_parse → aligned_instances → text_parse` 三层叠加得到统一槽位字典（后者覆盖前者）；② subtype 解析：recipe 显式 → 模板内按 input_mode 匹配 → 第一个可用 subtype；③ 用 `DIM_SHORT` 缩名 + 全局序列生成 `question_id`（如 `attr_multi_0001`）；④ 按 `prompt_pattern` 渲染 prompt_draft，失败则回退到 `base_prompt_draft`；⑤ 把模板里的 `target_plan / preserve_plan / dimension_isolation / evaluator_plan` 拷过来填实 |
| **产物** | `benchmark_dataset/question_plans.jsonl`，每行一条 `QuestionPlan` |
| **关键字段** | `question_id` / `recipe_id` / `dimension` / `input_mode` / `subtype` / `difficulty` / `semantic_rarity` / `contrastive_pair_id` + `contrastive_role` / `input_plan.required_images[].role` + `source_preference[]` / `target_plan.operation` + `expected_final_state` / `preserve_plan[]` / `dimension_isolation.forbidden_words` + `camera_constraint` / `evaluator_plan.E_target/P_constraints/C_criteria/tools` / `prompt_draft` |
| **下游消费者** | Step 4（构图）、Step 6（prompt 定稿）、Phase 3 evaluator |
| **工程用途** | 把单条 recipe 翻译成 Phase 3 可直接吃的「设计图」。Phase 3 不再回头解析自然语言 prompt |
| **论文用途** | 论证「评测目标 / 保持约束 / 维度隔离」三件套是题目的一等公民 |

**字段释义**：

- 三层槽位融合的目的：图像看到的真实主体 + 文本想要的目标变化 + 对齐后的 instance_id 一次性拼齐。
- `evaluator_plan.tools`：Phase 3 该题需要调用的评测器集合（grounding / segmentation / tracking / vlm / clip / flow 等），决定每题的工具栈预算。
- `prompt_draft` 只是占位与 QC 参考，**不是** Phase 3 给模型看的 prompt；最终 prompt 由 Step 6 写定。

**为什么这样设计**：

- **三层叠加 image > aligned > text 后者优先**：体现「意图 > 对齐 > 现状」原则——最终 prompt 服务于意图，所以 text 解析优先级最高；image_parse 提供「图里实际有什么」作为视觉错位检验。
- **subtype 三级回退而非直接报错**：模板库不可能枚举所有真实分布的 subtype 组合，「宽松匹配」比「严格匹配整桶失败」更利于 pilot 跑通。
- **question_id 用 DIM_SHORT + mode_short 而非哈希**：人工排查时一眼能从 ID 看出维度与模式（`attr_multi_0001` 即属性多图题）；grep / 分桶统计也按前缀零开销。
- **evaluator_plan 必须在 Phase 2 落定**：Phase 3 如果反向解析自然语言 prompt 推工具集，会引入 NLU 误差且不可复现。把工具集编码进 evaluator_plan 是让 Phase 3 变成纯执行器的关键。
- **prompt_draft 不能当 final**：draft 并未「看图说话」过，首帧实际状态与模板槽位可能出入（模板写 a woman、首帧实为两人），必须经 Step 6 VLM 对齐后才能用。

### Step 4 · `construct_inputs.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `question_plans.jsonl` + Phase 1 的 `manifest_clean.jsonl`（首帧路径）+ `assets.jsonl`（参考资产）+ `reference_bank/{asset_type}/*.jpg` |
| **处理逻辑** | ① 用 `_ROLE_TO_ASSET_TYPE` 把题目里的角色映射成 Phase 1 的 asset_type（target_subject→subject、attribute_reference→attribute、scene_reference→inpainted/original 等）；② 单图首帧直接复用 manifest 中 `image_path`；③ 多图按 `reference_asset_ids` 取资产，同角色多个候选时挑 `quality_score` 最高那张；④ `source_preference` 链式回退：`tip_derived_reference → t2i_generated → external`，回退到 T2I 时调用 `Phase2SiliconFlowClient.call_t2i`（默认 Kwai-Kolors/Kolors，1024×1024）；⑤ 双轨产物：主产物用 `resize_long_edge(..., enlarge=True)` 强制等比放大到 `long_edge=854`（480P 长边，保原比例）作为 evaluator 真值；同时调 `to_16x9_720p()` 产出严格 854×480 的 `<name>_16x9.png` 伴生件供 I2V 生成模型调用（resize / center crop / letterbox 三策略按原比例自动选） |
| **产物** | `benchmark_dataset/images/{question_id}.png`（首帧）+ `benchmark_dataset/references/{question_id}__{role}.png`（多图）+ `benchmark_dataset/input_assets_manifest.jsonl` |
| **关键字段** | `assets[].asset_id / role / path / source_type / source_ref_id / quality.identity_visibility / crop_leakage_risk / resolution_ok` |
| **下游消费者** | Step 5（QC）、Step 7（数据集导出）、Phase 3（模型输入） |
| **工程用途** | 把抽象的 question plan 落实成模型真正能吃的图像，并打上质量与来源标签 |
| **论文用途** | 数据集 sample 图谱；统计真实 vs 合成 vs 外部素材的占比 |

**字段释义**：

- `source_type` 三选一：`tip_derived_reference`（来自 TIP 首帧的 crop / 原图）/ `t2i_generated`（SiliconFlow 现合成）/ `external`（外部补充，正式版启用，需 provenance）。
- `quality.crop_leakage_risk`：背景泄露风险等级（low/medium/high），重点防「看背景就反推出空间关系」的捷径。
- T2I 客户端不可用或调用失败时，仅在 `quality.notes` 标记 `t2i_failed`，不在本步抛异常，由 Step 5 QC 决定是否丢弃。
- 路径约定：单图 = `images/{qid}.png`；多图 reference = `references/{qid}__{role}.png`，role 直接编进文件名便于人工核验。

**为什么这样设计**：

- **单图直接复用 TIP 首帧而非合成**：这是 benchmark 真实性的根基——单图任务的「起始状态」应当是真实视频首帧，否则评测的就不是「模型对真实首帧的处理能力」，与论文论点错位。
- **链式回退 tip → t2i → external 按「reviewer 可审计度」递降**：TIP 可追到原视频帧（最强）、T2I 可追到 prompt+seed（中等）、external 需单独 provenance（最弱）；把审计成本作为优先级权重而不是单看视觉质量。
- **long_edge=854 + PNG**：854 是 480P 标准长边（854×480，近似 16:9），与 CogVideoX-480 / Open-Sora 480P / SVD 等主流 I2V 模型 480P 推理档位对齐；相较 720P 可显著降低 Phase 3 推理开销。`resize_long_edge(..., enlarge=True)` 强制把低分辨率上游图（如 TIP-I2V 224×126）等比放大到 480P 长边，承担插值噪声以换取下游尺寸一致；PNG 无损避免 JPEG 边缘 ringing 伪影干扰 grounding 评测器。
- **双轨产物（`<name>.png` + `<name>_16x9.png`）**：I2V 生成模型多有硬编码的 16:9 输入约束，但 benchmark 首帧频繁出现正方形 / 竖屏 等非 16:9 比例。主产物保留原始等比供 evaluator P 维度作为真值（零形变、零裁剪、零黑边）；16:9 伴生件 `to_16x9_720p()` 按比例自动选取 resize / center crop / letterbox 三策略，供生成器调用。为什么竖屏选 letterbox 而非 crop：crop 竖屏会丢 56% 内容含主体，letterbox 仅带黑边且 SVD / CogVideoX 训练数据中本就含大量 letterbox 样本。
- **T2I 失败不抛异常**：流水线 8 步，任一步在单题失败如果中断会让上游全部白跑；让 QC 统一裁决，整体吞吐更稳。

### Step 5 · `verify_inputs.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `question_plans.jsonl` + `input_assets_manifest.jsonl` + 对应 PNG + 模板里声明的 `qc_checks[]` |
| **处理逻辑** | ① 对每张输入图按模板顺序调用 `Phase2SiliconFlowClient.call_vqa_structured`，VLM 返回 `{answer:bool, confidence:float, rationale:str}`；② 用 `_HARD_CHECK_NAMES` 区分硬检查与软检查；③ 三态聚合 `_aggregate_status`；④ 双队列分流 |
| **产物** | `benchmark_dataset/qc_reports/{question_id}.json` + `qc_failed_to_retry.jsonl` + `manual_review_queue.jsonl` |
| **关键字段** | `QCReport.qc_status ∈ {pass, fail, needs_manual_review}` / `checks[].{name, answer, confidence, hard_check}` / `risk_flags[]` |
| **下游消费者** | Step 6（仅 pass 进入定稿）、Step 7（导出过滤）、人工复核 |
| **工程用途** | 用结构化 VQA 给输入图把关，避免不合规图被带进 Phase 3 |
| **论文用途** | 论证数据集质量；可报告「通过率 / 软失败比例 / 失败模式分布」 |

**字段释义**：

- `_HARD_CHECK_NAMES`：11 项强制检查，包括 `has_target_subject_visible / single_target_subject / all_required_subjects_visible / target_subject_static / no_action_already_in_progress / no_motion_blur_present / interaction_not_yet_started / no_target_attribute_already / scene_has_subject_room / resolution_ok` 等。
- 三态聚合规则：
  - 任一 hard check `answer=False` 且 `confidence ≥ 0.7` → `fail`（写入 retry 队列）；
  - 任一 check `confidence < 0.7` → `needs_manual_review`（写入人工队列，**默认不进 Phase 3**）；
  - 全部通过 → `pass`。
- `risk_flags`：从 construct 阶段继承的标签（如 `t2i_failed`）+ 本步发现的风险，会被 Step 7 写进最终样本。

**为什么这样设计**：

- **用 VLM 而非规则匹配**：题目合规性是语义级判断（「主体是否还静止」「目标属性是否还没出现」），传统 CV 规则覆盖不全；VLM 是当前能给出语义级判定的最低成本工具。
- **confidence ≥ 0.7 才算 fail，<0.7 进人工**：VLM 在边界样本上有抖动，把「明确否决」与「模糊判断」分开——前者直接 retry（换图），后者换图也大概率再抖动，人裁决一次的边际成本更低且终局。
- **HARD_CHECK 集中在常量**：哪些 check 一票否决是评测设计的一等决策，模板作者不应能任意提升某个 check 的权重；集中常量也避免拼写错误造成「应阻断但被默认软处理」的静默退化。
- **反向检查（no_target_attribute_already 等）**：防「题目自我作弊」——若首帧已含目标终态，Static Copy baseline 会跟正常模型并列高分，anti-shortcut 实验直接破产。
- **needs_manual_review 默认不进 Phase 3**：multi-image 维度稀缺，但与其放进去拉低可信度不如等人裁决。reviewer 关心「过滤了多少」而非「保留了多少」。

### Step 6 · `finalize_prompts.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `question_plans.jsonl` + `qc_reports/`（仅取 pass）+ `input_assets_manifest.jsonl` + 首帧 PNG + 模板 `prompts/prompt_polish.txt` |
| **处理逻辑** | ① 用 VLM 给真实首帧生成中性 caption；② 把 caption + question_plan 的 target/preserve/isolation 拼进 polish 模板，调 LLM 要求返回 JSON `{i2v_prompt, reasoning}`；③ 解析阶段先剥 fenced ``` 再做括号扫描提取 JSON；④ 双校验：禁词命中即失败，字数常规 8–25 词、interaction 维度允许到 30 词；⑤ 最多重试 N 次仍不合格则回退到 `prompt_draft` 并标记 `prompt_polish_fallback` |
| **产物** | `benchmark_dataset/prompts/final_prompts.jsonl` |
| **关键字段** | `FinalPromptEntry { question_id, i2v_prompt, polish_attempts, used_fallback, vlm_caption }` |
| **下游消费者** | Step 7（数据集导出）、Phase 3（模型输入） |
| **工程用途** | 看着真实首帧把 prompt 写定，避免 prompt 与起始状态错位 |
| **论文用途** | 附录样例题；展示 prompt 风格贴近真实用户表达 |

**字段释义**：

- `i2v_prompt`：8–25 词英文（interaction 可到 30 词），多图时必须出现 `image 1 / image 2` 或 `the reference X` 这样的显式指代。
- `polish_attempts`：实际尝试次数，体现 LLM 一次过率。
- `used_fallback=true` 表明该题最终 prompt 是 draft，会在 risk_flags 留痕；Step 7 仍可放行，但人工复核优先。
- 禁词来自 `dimension_isolation.forbidden_words`，例如 motion 题禁 `pan/zoom`、spatial 题禁 `move/shift/reposition`。

**为什么这样设计**：

- **VLM caption + LLM polish 两步分工**：VLM 优于看图但弱于复杂格式约束，LLM 反之；合并一步让 VLM 直接出 prompt 会让 forbidden_words / 字数 / reference 指代难以管控，分工各取所长。
- **强制 LLM 返回 JSON**：裸文本下 LLM 常自作主张加引号、解释、markdown 标题；JSON 强约束让解析端能严格校验，同时保留 `reasoning` 字段便于事后审查「模型为什么这么改」。
- **字数 8–25 / interaction 30**：TIP-I2V 真实 prompt 中位数约 12 词，8–25 覆盖 ≈80% 真实分布；interaction 题需描述「主体 A + 主体 B + 因果链」，压到 25 词会丢信息（如 `the man hands the cup to the woman` 压成 `man hands cup` 会让 evaluator 定位不到 woman）。
- **禁词命中即重写而非软警告**：dimension_isolation 是评测可信度的最后防线——motion 题出现 pan 会让模型用相机平移伪装主体位移、spatial 题出现 move 会让两维度边界丢失；重写是把设计文档里的硬规则落到可执行约束。
- **fallback 仍允许 export**：流水线已把 QC pass 的样本送到这一步，因 polish 失败丢弃整题会让上游 4 步白费；标记 `prompt_polish_fallback` 让审计能统计 fallback 比例并触发后续修复。

### Step 7 · `export_dataset.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `question_plans.jsonl` + `input_assets_manifest.jsonl` + `qc_reports/*.json` + `final_prompts.jsonl` |
| **处理逻辑** | ① 以 `question_id` 为主键四源 join；② 仅保留 `qc_status=pass` 且 `i2v_prompt` 非空且 manifest 非空的样本；③ 多图质量按 worst-case 聚合：`crop_leakage_risk` 取 max、`identity_visibility` 取 min、`scale_compatibility` 由 bbox 面积比折算后取 min；④ 拼装 `BenchmarkSample`；⑤ 按维度落到 7 个分文件并同步写入 manifest |

| **关键字段（BenchmarkSample）** | `question_id` / `dimension` / `input_mode` / `subtype` / `difficulty` / `semantic_rarity` / `contrastive_pair_id` + `contrastive_role` / `input_images[]` / `i2v_prompt` / `metadata`（**target_subjects** + **reference_subjects** + **expected_change** + **preserve_constraints** + **evaluator_hints**） / `source_trace`（**recipe_id** + **source_type** + **phase1_sample_ids** + **phase1_asset_ids**） / `qc.{status, risk_flags}` / 多图额外字段 `reference_assets[]` + `multi_reference_quality` |

| **工程用途** | 把四条流水线汇聚成最终数据集；保证 Phase 3 拿到的每条样本都自洽 |
| **论文用途** | 数据集发布物；reviewer 可凭 `source_trace` 直接验证每条样本 provenance |

**字段释义**：

- `metadata.expected_change`：`{type, target_relation/target_attribute_after/...}`，Phase 3 E 评估的判别目标。
- `metadata.preserve_constraints`：Phase 3 P 评估的检查清单（哪些 instance / 属性 / 背景 / 镜头不得变）。
- `metadata.evaluator_hints.tools`：Phase 3 工具调度的硬约束；`primary_metric` 决定主分聚合口径。
- `source_trace.phase1_sample_ids` 与 `phase1_asset_ids`：每条样本可追溯到 TIP 原始 sample 与 reference_bank 资产。
- `multi_reference_quality`：多图样本的 worst-case 质量画像，是 Step 8 直方图的输入。

**为什么这样设计**：

- **以 question_id 落盘 join 而非内存串接**：四个上游模块各自独立写 jsonl，落盘 + 文件 join 是最稳的串接方式——任一步崩溃可单独重跑该步，前后产物保留可复查。
- **三重门控（pass + 非空 prompt + 非空 manifest）**：三种空值各自代表不同失败模式（质量否决 / polish 失败 / 构图失败），在 export 阶段就过滤问题样本，比让 Phase 3 模型跑完才发现节省至少一个量级的算力。
- **多图质量按 worst-case 聚合**：评测可信度的关键是「任何一张参考的弱点都会被模型利用」，max / 平均聚合会掩盖「看似合格但有薄弱点」的多图样本。
- **phase3_manifest 是 BenchmarkSample 的视图**：保证两份产物 schema 同源，manifest 字段调整时直接从 Sample 派生，避免「加了字段但忘了同步 manifest」的 schema drift。
- **按 dimension 分文件**：Phase 3 evaluator 按维度调度，分文件让「只跑某维度」零开销；论文统计按维度切片也直接 `wc -l` 出数。
- **source_trace 必带 phase1_sample_ids 与 phase1_asset_ids**：reviewer 一抽样就要能回到 TIP 原视频与原资产去验证「这道题确实源自真实分布」；少任何一个 ID，「provenance 反查」链条就断了。

### Step 8 · `audit_phase2.py`

| 项 | 说明 |
|---|---|
| **原始输入** | `quota_plan.json` + `sampled_recipes.jsonl` + `question_plans.jsonl` + `qc_reports/*.json` + `samples/*.jsonl` + `phase3_manifest.jsonl` |
| **处理逻辑** | 六类校验：① 行数一致（samples 总和 == manifest 行数）；② 按 bucket_id 算配额 gap；③ 各维度 single/multi/subtype/difficulty/rarity 分布；④ contrastive 配对覆盖（A 与 B 是否成对存在）；⑤ 多图质量直方图（crop_leakage / identity_visibility / scale_compatibility）；⑥ QC 状态直方图（pass / fail / needs_manual_review 比例） |
| **产物** | `benchmark_dataset/dataset_card.md`（六节统计 + 风险摘要 + 验收清单） |
| **关键字段** | 各维度实际产出数、contrastive pair 覆盖率、reference quality 分布、QC 通过率 |
| **下游消费者** | 项目 owner、Phase 3 启动前的 gating |
| **工程用途** | 出货前最后体检；不修数据，只生成数据卡，由人决定是否放行进 Phase 3 |
| **论文用途** | 数据集 card；论文表 1（dataset statistics） |

**字段释义**：

- 各维度实际产出数：与 `quota_plan` 对照，缺口必须显式列出，禁止偷减。
- contrastive pair 覆盖率：要求所有 `contrastive_pair_required=true` 的桶都达成 A/B 配对。
- reference quality 分布：低于阈值的多图样本会在 dataset_card 显式预警。

**为什么这样设计**：

- **独立见证人而非与 export 共用代码**：审计逻辑与导出逻辑共用函数会让 bug 互相隐藏（同一个错误字段在两边都按错规则处理仍「自洽」通过）；分离两者才能在自检失败时给出可信信号。
- **不自动修复**：自动修复的本质是「猜测意图」，但每类缺口背后根因不同（数据不够 / 阈值太严 / 上游 bug）；把决策权留给人比让脚本「按默认策略修复」安全得多。
- **六类校验缺一不可**：六个维度对应六类风险——行数 → 完整性、配额 gap → 代表性、维度统计 → 均衡性、contrastive 配对 → 实验设计完整性、质量直方图 → 评测可信度、QC 直方图 → 成本可控性。漏任一项都会在 Phase 3 才暴露问题。
- **dataset_card.md 用 Markdown 而非 JSON**：论文素材直接 copy/paste，CI 可读，git diff 友好；机器需要的关键数据由 export 阶段已提供。

## 四、Phase 3：模型评测

> Phase 3 消费 `phase3_manifest.jsonl`，对待评 I2V 模型生成视频并打分，输出 `runs/eval/{model_name}/`。

### Step 1 · `generate_videos.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/generations/<question_id>.mp4` + 每条 `GeneratedVideo` 元数据 |
| **关键字段** | `status`（success / failed / unsupported_input_mode）/ `generation_params`（seed / num_frames / fps / resolution）/ `errors` |
| **下游消费者** | Step 2（预处理）、Step 5（评测器）、报告 |
| **工程用途** | 调用待评模型；失败的样本要记录但不能中断全局评测 |
| **论文用途** | 模型对比的原始素材；`unsupported_input_mode` 计数可揭示哪些模型不支持多图 |

**字段释义**：

- `status`：三状态：
  - `success`：生成成功
  - `failed`：生成失败（超时 / OOM / API 错误）
  - `unsupported_input_mode`：模型不支持多图输入。该状态在论文中作为能力诊断依据，不计为模型质量问题。
- `generation_params`：生成参数快照（seed / num_frames / fps / resolution）。保证可复现。
- `errors`：失败原因。使质量问题可以区分于能力问题。

### Step 2 · `preprocess_videos.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/normalized_videos/` + 抽帧目录 |
| **关键字段** | `normalized_video_path` / `frames_dir` / `num_frames` / `fps` / `resolution` |
| **下游消费者** | Step 3 / 4（grounding & tracking）、所有 evaluator |
| **工程用途** | 把不同模型的输出统一到固定 fps / 分辨率；保证后续评测器可比 |
| **论文用途** | 实验设置章节中对"评测时帧率统一"的实施依据 |

**字段释义**：

- `normalized_video_path`：统一到固定 fps / 分辨率后的视频路径。所有 evaluator 只读这个。
- `frames_dir`：抽帧后的帧目录（供 grounding / tracking / VLM 不重复读取）。
- `num_frames` / `fps` / `resolution`：统一后的元信息，保证不同模型间评测可比。

### Step 3 · `grounding.py`

| 项 | 说明 |
|---|---|
| **产物** | 每帧的 bbox / mask + grounding confidence |
| **关键字段** | `target_subject_id` / `bbox_per_frame[]` / `mask_per_frame[]` / `grounding_confidence` / `tool_status` |
| **下游消费者** | Step 4（tracking）、所有 evaluator |
| **工程用途** | 把 metadata.target_subjects 在视频中真正"找到"；失败时标记 `tool_uncertain`，不简单给 E=0 |
| **论文用途** | 工具置信度报告；解释 benchmark 的可信度边界 |

**字段释义**：

- `target_subject_id`：被定位的主体在 metadata 中的 ID。来自 Phase 2 的 target_subjects。
- `bbox_per_frame[]` / `mask_per_frame[]`：逐帧的几何位置。Spatial / Motion evaluator 的原材料。
- `grounding_confidence`：逐帧检测置信度。低于阈值该帧不计入评测。
- `tool_status`：`valid` / `low_confidence` / `tool_uncertain` / `invalid_input`。**不能简单将 tool_uncertain 设为 E=0。**

### Step 4 · `tracking.py`

| 项 | 说明 |
|---|---|
| **产物** | 主体轨迹 / mask tube + tracking confidence |
| **关键字段** | `target_tube` / `displacement_curve` / `tracking_confidence` / `lost_ratio` |
| **下游消费者** | Motion / Spatial / Action / Interaction evaluator |
| **工程用途** | 给位移类与动作类维度提供时序证据；丢失比例进入 tool_confidence |
| **论文用途** | 失败案例诊断；展示 tracking 在生成视频上的有效边界 |

**字段释义**：

- `target_tube`：主体跨帧的 mask tube（三维轨迹）。作为 Motion / Action evaluator 的输入。
- `displacement_curve`：位移曲线（时间 × 位置）。Motion 维度的方向 / 速度判定依据。
- `tracking_confidence`：跨帧 tracking 置信度。进入 tool_confidence 聚合。
- `lost_ratio`：被跟丢的帧占比。过高则样本转 `tool_uncertain`。

### Step 5 · `evaluators/{dimension}.py`

| 项 | 说明 |
|---|---|
| **产物** | 每条样本的 `EvalResult`（暂存内存或临时 JSONL） |
| **关键字段** | `E` / `P` / `C` / `S` / `aggregation=exec_gated` / `lambda=0.6` / `tool_status` / `tool_confidence` / `failure_modes[]` / `auxiliary` |
| **下游消费者** | Step 6（统一打分写盘）、Step 8（聚合）、Step 9（人类对比） |
| **工程用途** | 七维度各自的核心评测逻辑；E/P/C 必须独立保存 |
| **论文用途** | 雷达图、E/P/C 分解柱状图、失败模式案例图 |

**字段释义**：

- `E`（Execution）：prompt 要求的变化是否被执行。默认 \[0,1]。
- `P`（Preservation）：保持约束是否被遵守。默认 \[0,1]。
- `C`（Coherence）：过程是否平滑、无闪烁 / 跳变 / identity drift。默认 \[0,1]。
- `S`：最终样本分数（由 Step 6 计算并写入）。
- `aggregation=exec_gated` / `lambda=0.6`：记录评分公式与权重，保证可复现。
- `tool_status` / `tool_confidence`：工具状态 + 置信度。`tool_uncertain` 不进入主表聚合。
- `failure_modes[]`：本题命中的失败模式标签集（如 `static_copy_like` / `wrong_direction`）。论文失败案例图的分类依据。
- `auxiliary`：辅助字段（如原始 iou 值 / 位移距离），供后续分析。

### Step 6 · `scoring.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/per_sample_scores.jsonl` |
| **关键字段** | 应用 `S = E * (0.6P + 0.4C)` 后的最终样本分；`tool_uncertain` 时 `S=null` |
| **下游消费者** | Step 8（聚合）、Step 9（人类一致性） |
| **工程用途** | 统一执行门控公式；防止"静态复制 + 高 P/C"刷分 |
| **论文用途** | 主表分数列；门控公式有效性的实验依据 |

**字段释义**：

- 门控公式：`S = E * (0.6 * P + 0.4 * C)`。乘法设计使“静态复制”这类退化解拿不到高分。
- `tool_uncertain` 样本：`S = null`，不进入聚合，但记录在 tool_uncertainty_report。
- `S` 范围：\[0,1]，与 E/P/C 同量纲。

### Step 7 · `baselines.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/baseline_<name>/per_sample_scores.jsonl` |
| **关键字段** | static_copy / random_motion / global_filter / camera_pan_cheat 四类 baseline 的 E/P/C/S |
| **下游消费者** | Step 8（聚合到对照表）、论文 anti-shortcut 分析 |
| **工程用途** | 验证评测器不会奖励退化解；Static Copy 总分应 < 0.10 |
| **论文用途** | 评测器可信度的关键证据章节；reviewer 最关心的 anti-shortcut 表 |

**字段释义**：

- `static_copy`：把首帧复制为静态视频。**预期总分 < 0.10**，否则说明门控失效。
- `random_motion`：随机平移 / 随机背景变化。验证模型是否能区分“乱动”与“按指令动”。
- `global_filter`：整图调色 / 滤镜。**不能在 background_dynamics 以外拿高分**。
- `camera_pan_cheat`：镜头平移伪装主体在动。**不能在 motion / spatial 拿高分**。

### Step 8 · `aggregate.py`

| 项 | 说明 |
|---|---|
| **产物** | `dimension_scores.json` + `input_mode_scores.json` + `overall_report.json` + `failure_diagnostics.json` + `tool_uncertainty_report.json` |
| **关键字段** | 按 dimension / input_mode / subtype / difficulty / semantic_rarity / source_type / contrastive_pair 七种分组聚合的 S / E / P / C；`tool_uncertain_rate` |
| **下游消费者** | Step 10（报告）、论文图表 |
| **工程用途** | 把样本级分数聚合到论文表所需粒度；同时提供 single-only / multi-only 视角 |
| **论文用途** | 主表（每个维度的 S/E/P/C）；多图 vs 单图对比图；难度桶差异图 |

**字段释义**：

- `dimension_scores.json`：七维度的 S / E / P / C 聚合。论文主表。
- `input_mode_scores.json`：`S_dim_single` 与 `S_dim_multi` 独立报告，避免不支持多图的模型被显著低估。
- `overall_report.json`：总表。含混合 / single-only / multi-only 三种口径。
- `failure_diagnostics.json`：按 failure_modes 聚合。`invalid_input` 样本仅出现在该报告中，不计入 mean。
- `tool_uncertainty_report.json`：工具不确定样本的占比与分布。论文可信度部分的补充表。
- 聚合分组键：dimension / input_mode / subtype / difficulty / semantic_rarity / source_type / contrastive_pair 七种。

### Step 9 · `human_validation.py`

| 项 | 说明 |
|---|---|
| **产物** | `human_subset_manifest.jsonl` + `human_annotations_template.csv` + `human_correlation_report.json` |
| **关键字段** | 抽样的 question_id 子集；标注表（human_E / human_P / human_C / failure_modes / artifact_severity）；Spearman / Kendall 相关性 |
| **下游消费者** | 论文人类一致性章节、Interaction 维度（默认更高比例） |
| **工程用途** | 验证自动评测器与人类判断的一致性；按 artifact severity 分桶可揭示工具失效边界 |
| **论文用途** | benchmark 可信度的关键章节；通常是 reviewer 最关心的“自动指标是否对得上人评” |

**字段释义**：

- `human_subset_manifest`：抽样题集。正式版每维度 50–100 题；pilot 每维度全量 20 题。
- `human_annotations_template.csv`：给人工标注员填写的表（human_E / human_P / human_C / failure_modes / artifact_severity）。
- `human_correlation_report`：Spearman / Kendall 相关性 + 标注者间一致性。
- 分桶报告：按 artifact severity 分桶，揭示工具在生成视频上的有效边界。
- Interaction 维度：默认人工抽样比例高于其它维度（其评测器置信度较低）。

### Step 10 · `report.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/visualizations/` + 汇总 markdown / pdf |
| **关键字段** | 七维度雷达图、E/P/C 柱状图、single vs multi 对比、与 VBench-I2V 排名差、Static Copy 表、人类一致性表、典型失败案例图 |
| **下游消费者** | 论文作图、对外发布 |
| **工程用途** | 一键生成论文所需图表；统一格式便于多模型横向对比 |
| **论文用途** | 论文主图、补充材料、leaderboard 页面 |

**字段释义**：

- `七维度雷达图`：跨维度能力谱。默认轴为 S，可切换为 E / P / C。
- `E/P/C 柱状图`：揭示模型是“执行弱”还是“保持弱”。
- `single vs multi 对比`：同一模型在两种 input_mode 下的差异。
- `与 VBench-I2V 排名差`：证明本 benchmark 与质量型 benchmark 不同步，提供互补信息。
- `Static Copy 表`：anti-shortcut 主证据。
- `人类一致性表`：按维度列 Spearman / Kendall。
- `典型失败案例图`：按 failure_mode 分组的选例，附充原始首帧 + 生成视频抽帧。

## 六、一句话速览

- **Phase 1 产出"做题用的原料和说明书"**：先验包、参考资产库、可采样 recipe、维度兼容矩阵、维度隔离约束。（Step 1–5 基础分析 + Phase 1 增强 6 步）
- **Phase 2 产出"标准化考卷"**：每道题带输入图、prompt、目标变化、保持约束和评测指引。
- **Phase 3 产出"考试结果与可信度证据"**：每条样本的 E/P/C/S、维度聚合、baseline 对照、人类一致性。

三个阶段的产物共同构成**可追溯、可复现、可审计**的 I2V 组合性评测闭环。
