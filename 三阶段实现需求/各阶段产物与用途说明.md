## 二、Phase 1：先验数据准备

> Phase 1 的目标是把原始 TIP-I2V 数据转化为结构化的 phase1 产物，为 Phase 2 提供清洁清单、结构化解析、参考资产、先验包与可采样 recipe。
>
> **当前实现位置**：`tip_i2v_data_analysis/` 目录。前置步骤 (Step 1–5) 位于 `src/step*`，Phase 1 增强步骤位于 `src/phase1/`。
>
> **CLI 入口**：`python main.py --step <name> --config configs/config.yaml`

### Step 1 · `step1_manifest.py`（`--step manifest`）

| 项 | 说明 |
|---|---|
| **产物** | `{manifest_dir}/manifest_clean.jsonl`（仅 `status=ok`）+ `images/<sample_id>.jpg` + `bad_samples.jsonl` |
| **关键字段** | `sample_id` / `clean_prompt`（剥离 Pika 参数后的自然语言）/ `pika_camera` / `pika_motion` / `image_path` / `status` |
| **下游消费者** | Step 2（图像解析）、Step 3（文本解析）、Phase 1 增强步骤 |
| **工程用途** | 给所有后续步骤提供唯一的样本主键与首帧文件；分离平台参数与自然语言意图，避免污染解析 |
| **论文用途** | 在数据章节中报告"我们使用 N 条 TIP-I2V 样本，清洗成功率 X%"；`pika_camera` 分布可作为运镜真实先验的直接证据 |

**字段释义**：

- `sample_id`：样本唯一主键（`tip_XXXXXX`）。所有后续步骤都用它做 join key，绝对不允许变化。
- `clean_prompt`：剥离 `-camera`、`-motion` 等 Pika 平台参数后的纯自然语言。用于 LLM 文本解析、prompt 模板构造。
- `pika_camera` / `pika_motion`：从原始 prompt 中分离出的平台参数（pan/zoom/tilt 等）。**只用于统计运镜先验**，不进入 prompt 解析的"用户意图"。
- `image_path`：首帧图像在本地磁盘的相对路径。Step 2 和 Phase 1 多图素材抽取的输入。
- `status`：清洗状态（`ok` / `bad_image` / `bad_prompt`）。Step 2 之后只处理 `ok`，`bad_*` 进入 `bad_samples.jsonl` 仅用于审计。

### Step 2 · `step2_image_analysis.py`（`--step image`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/image_analysis/image_parse.jsonl` |
| **关键字段** | `subjects[]`（含 category / description / position_in_frame / attributes / is_animate / visibility）/ `subject_relations[]` / `background` / `camera_baseline` |
| **下游消费者** | Step 4（联合分析）、Phase 1 patch（补齐 bbox / instance_id）、Phase 1 refbank（资产抽取） |
| **工程用途** | 通过 VLM 把首帧"看懂"成结构化字段；为后续 mock_geometry 提供 position_in_frame |
| **论文用途** | 报告"首帧主体数 / 场景类别 / shot type"分布；论证多图任务的真实视觉先验来源 |

**字段释义**：

- `subjects[]`：首帧中 VLM 识别的所有主体。每个主体含：
  - `category` / `description`：主体类别与描述，供 Phase 1 align 做名称软匹配
  - `position_in_frame`（9 宫格：center / upper-left / ...）：Phase 1 patch 据此反推归一化 bbox
  - `attributes`（颜色 / 服饰 / 朝向）：属性维度的题目原料
  - `is_animate` / `visibility`：用于推断 tracking_feasibility 与 segmentation_quality
- `subject_relations[]`：首帧主体两两之间的空间关系（on / beside / holding 等）。Spatial / Interaction 维度的真实先验。
- `background`：场景类别 + 室内外 + 主要元素。Background Dynamics 题目的判定基线。
- `camera_baseline`：首帧的镜头基线（shot type、视角）。View 维度的"起点"。

### Step 3 · `step3_text_analysis.py`（`--step text`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/text_analysis/text_parse.jsonl` |
| **关键字段** | `involves_*` 六个布尔字段 / `attribute_change_slots` / `action_slots` / `motion_slots` / `spatial_relation_slots` / `background_change_slots` / `camera_motion_slots` |
| **下游消费者** | Step 4（联合分析）、Phase 1 patch（从 involves_* 推导 primary_dimension） |
| **工程用途** | 通过 LLM 把自然语言归一为结构化槽位；Phase 1 patch 据此进行维度路由 |
| **论文用途** | 真实分布章节的核心数据来源；解释为什么每个维度的常见概念是这些 |

**字段释义**：

- `involves_attribute_change` / `involves_action` / `involves_directed_motion` / `involves_spatial_relation_change` / `involves_background_change` / `involves_camera_movement`：六个布尔标记，Phase 1 patch 按优先级推导 primary_dimension。
- `*_slots`（六种）：维度专属的结构化槽位。例：
  - `attribute_change_slots`：`target_subject` / `attribute_before` / `attribute_after`
  - `motion_slots`：`subject` / `direction` / `manner`
  - `spatial_relation_slots`：`subject_a` / `relation` / `subject_b`
  这些槽位直接喂给 Phase 1 recipes 与 Phase 2 的 question_plan。
- LLM prompt 模板已扩展第三层"维度路由与防泄漏"（`prompts/llm_text_parse.txt`），支持直出 `interaction_slots` / `primary_dimension` / `candidate_dimensions` / `forbidden_dimension_leakage` / `routing_reason`。

| **论文用途** | 提供维度共现矩阵、运镜先验等论文图表数据 |

### Phase 1 增强步骤（`src/phase1/`）

> 以下 6 个步骤是在 Step 1–5 基础上的增强，**不重跑 VLM/LLM**，通过启发式补齐 + 几何 mock + 对齐 + 资产抽取完成 Phase 1 闭环。
>
> 一键全跑：`python main.py --step phase1 --config configs/config.yaml`

### Step P1.1 · `patch_existing_outputs.py`（`--step patch`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/image_analysis/image_parse_v2.jsonl` + `{output_dir}/text_analysis/text_parse_v2.jsonl` |
| **关键字段** | 图像新增：`instance_id` / `bbox` / `segmentation_quality` / `tracking_feasibility` / `reference_potential`；文本新增：`primary_dimension` / `candidate_dimensions` / `forbidden_dimension_leakage` / `interaction_slots` / `routing_reason` |
| **下游消费者** | P1.2 align、P1.3 refbank、P1.5 recipes |
| **工程用途** | 在不重跑 VLM/LLM 前提下，启发式补齐 Phase 1 所需新字段；使用 mock_geometry（9 宫格→bbox）替代真实分割 |
| **论文用途** | 体现 benchmark 流水线的数据增量设计——增量补齐而非全量重跑 |

**核心机制**：

- **图像补丁**：从 VLM 输出的 `position_in_frame` 通过 9 宫格映射反推归一化 bbox；根据 `visibility` / `is_animate` 推断 `segmentation_quality`（high/medium/low/unavailable）与 `tracking_feasibility`（easy/medium/hard/infeasible）
- **文本补丁**：从 `involves_*` 六个布尔字段按优先级推导 primary_dimension（interaction > attribute > action > motion > spatial > background > view）；通过 LEAKAGE_TABLE 自动补齐 forbidden_dimension_leakage
- **instance_id** 命名约定：`inst_{sample_id}_{idx}`

### Step P1.2 · `align_instances.py`（`--step align`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/phase1/aligned_instances.jsonl` |
| **关键字段** | `aligned_subjects[]`（instance_id / image_subject_id / image_subject_name / text_subject_name / alignment_confidence / geometry）/ `unmatched_text_subjects` / `unmatched_image_subjects` / `overlap_ratio` / `evaluator_feasibility[]`（7 维各一条）/ `dimension_isolation` |
| **下游消费者** | P1.4 priors_enhance、P1.5 recipes、Phase 3 evaluator |
| **工程用途** | 把"prompt 提到的红球"对齐到"图像中的 inst_xxx_0"；为每个维度生成 tool_status 可评测性诊断 |
| **论文用途** | 论证 benchmark 样本是"图文对齐 + 评测可行"双约束筛选出来的 |

**字段释义**：

- `aligned_subjects[]`：文本-图像主体对齐结果。每条含 `instance_id`、`image_subject_id`、`image_subject_name`、`text_subject_name`、`alignment_confidence`（匹配置信度）、`geometry`（InstanceGeometry 字典）。
- `evaluator_feasibility[]`：7 维各一条，每条含：
  - `dimension`：维度名
  - `feasible`：是否可评测（bool）
  - `tool_status`：四态之一（valid / low_confidence / tool_uncertain / invalid_input）
  - `reasons`：不可评测原因
  - `required_tools`：需要的评测工具集
- `dimension_isolation`：含 `primary_dimension` / `forbidden_dimensions` / `leakage_risk_notes`
- `overlap_ratio`：图文主体匹配覆盖率（0–1）。低于阈值的样本仍保留但 recipes 可按此过滤。

### Step P1.3 · `reference_bank.py`（`--step refbank`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/phase1/assets.jsonl` + `reference_bank/{asset_type}/*.jpg` |
| **关键字段** | `asset_id` / `asset_type`（subject / attribute / object / scene_reference_original / scene_reference_inpainted）/ `usable_for[]` / `quality`（resolution / aspect_ratio / is_clean_background / has_visible_subject / quality_score / quality_flags）/ `provenance`（必填） |
| **下游消费者** | P1.5 recipes（多图 recipe 的 reference_asset_ids）、Phase 2 `construct_inputs.py` |
| **工程用途** | 多图题目的素材仓库——主体 crop、场景原图。provenance 字段保证每个 asset 可追溯到原始 TIP 首帧 |
| **论文用途** | 解决"多图模式无先验"的 reviewer 痛点；证明多参考输入来自真实 I2V 首帧拆解 |

**字段释义**：

- `asset_id`：`{sample_id}_{instance_id}_{suffix}` 格式的全局唯一 ID（suffix = subj/attr/obj）；场景类型为 `{sample_id}_scene_orig`。
- `asset_type`：五类资产（P0 阶段 `scene_reference_inpainted` 暂跳过）。
- `usable_for[]`：本 asset 适用的维度集。根据 asset_type 自动填充默认值。
- `quality`：含 resolution / aspect_ratio / is_clean_background / has_visible_subject / quality_score / quality_flags 六个子字段，决定该 asset 能否安全进入考题。P0 阶段基于 bbox 面积推断 quality_score。
- `provenance`（**必填**）：`source_sample_id` / `source_image_path` / `extraction_method`（mock_crop_9grid / original）/ `extraction_params` / `created_by` / `created_at`。audit 校验非空率必须 100%。

### Step P1.4 · `priors_enhance.py`（`--step priors2`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/phase1/priors/frequency_tiers.json` + `subject_pair_distribution.json` + `multi_reference_priors.json` + `compatibility_matrix.json` |
| **关键字段** | 频率三档（head/torso/long_tail）/ 主体共现频次 / 多图模式统计 / 7×7 维度兼容矩阵 |
| **下游消费者** | Phase 2 `build_quota.py`（采样比例）、`sample_recipes.py`（common/rare 分桶） |
| **工程用途** | 在 Step 5 prior_package 基础上补充 Phase 2 所需的配额先验；compatibility_matrix 作为维度隔离的参考 |
| **论文用途** | 数据驱动出题的核心证据；可视化为"共现矩阵 / 频率分桶"等论文图表 |

**四个产物释义**：

- `frequency_tiers.json`：概念频率三档——`head`（前 50%）/ `torso`（50%–85%）/ `long_tail`（后 15%）。对 subject category / action / attribute 分别分桶。
- `subject_pair_distribution.json`：多主体共现频次矩阵，key = `"{cat_a}::{cat_b}"`。Spatial / Interaction 多图组合合理性的判定。
- `multi_reference_priors.json`：多图模式统计——total_samples / multi_eligible_samples / multi_ratio / 各 asset_type 可用数量。
- `compatibility_matrix.json`：7×7 维度兼容矩阵，三态（compatible / leakage_risk / incompatible）。例如 attribute × motion = incompatible。

### Step P1.5 · `recipes.py`（`--step recipes`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/phase1/candidate_recipes.jsonl` |
| **关键字段** | `recipe_id` / `target_dimension` / `input_mode` / `source_type` / `base_prompt_draft` / `preserve_constraints` / `contrastive_spec` / `dimension_isolation` / `reference_asset_ids` / `expected_difficulty` |
| **下游消费者** | Phase 2 `sample_recipes.py`（**唯一题目入口**） |
| **工程用途** | 把"先验 + 资产 + 约束"打包成可直接采样的题目候选；保证 Phase 2 不会绕过 Phase 1 自己拼词表 |
| **论文用途** | 论文中"题目从哪里来"的直接答案；每条 recipe 的 source_type 可追溯，使 benchmark 可审计 |

**字段释义**：

- `recipe_id`：`recipe_{sample_id}_{dim}` 格式的唯一 ID。Phase 2 采样后 1:1 映射到 question_id。
- `target_dimension`：所属维度（DIMENSIONS_V2 七选一）。
- `input_mode`：`single_image` 或 `multi_image`。决策规则：view/background → single；spatial/interaction 且 ≥2 subject_asset → multi。
- `source_type`：四选一（observed_single_image / derived_single_image / derived_multi_reference / external_multi_reference_optional）。
- `base_prompt_draft`：`[dim={dim}] {clean_prompt}` 格式的 prompt 初稿。
- `preserve_constraints[]`：按维度自动填充默认保持约束（background / camera / identity / appearance / position / layout）。
- `contrastive_spec`：默认含 4 类 baseline（static_copy / random_motion / global_filter / camera_pan_cheat）。
- `dimension_isolation`：从 aligned_instances 继承的隔离约束。
- `reference_asset_ids[]`：multi_image recipe 关联的 asset_id 列表，可追溯到 reference_bank。
- `expected_difficulty`：easy / medium / hard。基于 primary_dimension + 主体数量 + alignment_confidence 推断。

### Step P1.6 · `audit.py`（`--step audit`）

| 项 | 说明 |
|---|---|
| **产物** | `{output_dir}/phase1/phase1_audit_report.md` |
| **关键字段** | 7 维度 recipe 计数、asset 计数与 provenance 非空率、各维度 single/multi 比例、tool_status 分布、isolation_violations |
| **下游消费者** | 项目 owner、合作者、Phase 2 启动前的 gating |
| **工程用途** | 进入 Phase 2 的门禁：如果某维度 recipe 数不够 pilot 的 3 倍，必须先回头补 |
| **论文用途** | 数据章节附录里的统计表；展示流水线的可复现与可审计性 |

**审计报告 7 节结构**：

1. **样本规模**：aligned 样本总数、image_parse_v2 / text_parse_v2 行数
2. **主维度路由分布**：每维度 recipe 数量统计
3. **7 维可评测性**：各维度 tool_status 分布（valid / low_confidence / tool_uncertain / invalid_input）
4. **资产 & provenance 完整性**：各 asset_type 数量 + provenance 非空率（期望 100%）
5. **Recipe 分布**：按 target_dimension / input_mode 交叉统计
6. **隔离有效性抽查**：dimension_isolation.forbidden_dimensions 违规样本数（期望 0）
7. **验收清单**：pass / fail 判定

## 三、Phase 2：Benchmark 数据集合成

> Phase 2 消费 Phase 1 产物（`candidate_recipes.jsonl` + `assets.jsonl` + `prior_package.json` + `compatibility_matrix.json` 等），产出最终 Benchmark 数据集 `benchmark_dataset/`，并提供 Phase 3 可直接消费的 `phase3_manifest.jsonl`。

### Step 1 · `build_quota.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/quota_plan.json` |
| **关键字段** | 每维度 / input_mode / subtype / difficulty / rarity 的目标计数；full=1400、pilot=140 |
| **下游消费者** | Step 2（采样）、Step 8（审计）、Phase 3 报告分组键 |
| **工程用途** | 把"7×200 题"展开成具体配额桶，决定每个桶要采多少 recipe |
| **论文用途** | 论文方法章节的"题目分布表"；解释 multi_image 比例为何这样设（数据驱动而非平均）|

**字段释义**：

- 每个配额桶由五个维度定义：`dimension` × `input_mode` × `subtype` × `difficulty` × `rarity`。
- `target_count`：该桶需要采样出多少题。full=1400、pilot=140 下的具体数字来自 Phase 1 prior_package。
- `difficulty`：Easy / Medium / Hard。各 40/40/20，判定规则见合作者版 §0.3。
- `rarity`：`common` / `rare`。80/20 划分，源于 prior_package 的 frequency_tiers。
- `contrastive_pair_required`：该桶是否需要成对采样（例如 attribute 维度 “red → blue” 与 “blue → red” 要成对）。

### Step 2 · `sample_recipes.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/sampled_recipes.jsonl` + `quota_unfilled_report.json` |
| **关键字段** | 采样后的 recipe 列表（含 contrastive_pair_id 配对）；缺口报告含每个未填满桶的原因 |
| **下游消费者** | Step 3（构 question plan） |
| **工程用途** | 在配额约束下从 candidate_recipes 抽取实际题目；不允许静默降级 input_mode |
| **论文用途** | 缺口报告可作为“benchmark 仍可扩展的方向”的依据 |

**字段释义**：

- `sampled_recipes`：从 candidate_recipes 中中选出的 recipe 集（中签后仍保留 recipe_id，供追溯）。
- `contrastive_pair_id`：同一对 contrastive 题共享的配对 ID。Phase 3 可按此计算模型在正反题上的一致性。
- `quota_unfilled_report`：每个未填满桶的原因（例如“multi_reference assets 不足”）。
- 采样原则：**不允许静默降级 input_mode**。多图桶不够时必须报警且在报告中明写。

### Step 3 · `build_question_plan.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/question_plans.jsonl` |
| **关键字段** | `input_plan`（required_images）/ `target_plan`（operation + expected_final_state）/ `preserve_plan` / `dimension_isolation`（forbidden_words / camera_constraint）/ `evaluator_plan`（E_target / P_constraints / C_criteria / tools）/ `prompt_draft` |
| **下游消费者** | Step 4（构图）、Step 6（prompt 定稿）、Phase 3 evaluator |
| **工程用途** | 题目的"设计图"——比 prompt 信息量大得多，让 Phase 3 不需要再猜 prompt 含义 |
| **论文用途** | 解释 benchmark 题目的结构化设计；展示“评测目标 / 保持约束 / 维度隔离”三件套 |

**字段释义**：

- `input_plan`：输入图计划。含 `required_images` 清单（单图 / 多图每张的 role + 来源）。Step 4 据此构图。
- `target_plan`：评测目标。含：
  - `operation`：要求发生的变换（move_left / change_color / zoom_in 等）
  - `expected_final_state`：Phase 3 期望看到的终态（如“s1 位于 s2 右侧”）
- `preserve_plan`：保持约束。列出哪些实例 / 属性 / 背景 / 镜头不得变。Phase 3 P 评估的检查清单。
- `dimension_isolation`：维度隔离双保险：
  - `forbidden_words`：prompt 中不得出现的词（防止泄漏其它维度）
  - `camera_constraint`：镜头是否必须静止（除 View 维度外的默认强约束）
- `evaluator_plan`：Phase 3 评测指引：
  - `E_target`：E 要评什么
  - `P_constraints`：P 要保什么
  - `C_criteria`：C 的判断点（闪烁 / 跳变 / identity drift）
  - `tools`：需要调用的评测器集
- `prompt_draft`：prompt 初稿。Step 6 打磨后才进入 final_prompts。

### Step 4 · `construct_inputs.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/images/` + `references/` + `input_assets_manifest.jsonl` |
| **关键字段** | 每张输入图的 `role` / `path` / `source_type`（tip_derived_reference / t2i_generated / external）/ `quality`（identity_visibility / crop_leakage_risk） |
| **下游消费者** | Step 5（QC）、Step 7（数据集导出）、Phase 3（模型输入） |
| **工程用途** | 将 Question Plan 落实成具体输入图；多图样本至少 2 张参考图，每张职责单一 |
| **论文用途** | 数据集 sample 图谱；展示真实 vs 生成 vs 外部素材占比 |

**字段释义**：

- `role`：该输入图在题目中的职责（首帧 / subject reference / attribute reference / scene reference / object reference）。多图样本中每张图职责必须唯一。
- `path`：输入图代号路径。Phase 3 递给模型的输入。
- `source_type`：生成来源，三选一：
  - `tip_derived_reference`：从 TIP 首帧拆出的真实资产
  - `t2i_generated`：T2I 生成补充的干净资产
  - `external`：外部补充（仅正式版启用，必记 provenance）
- `quality`：
  - `identity_visibility`：身份可識别度
  - `crop_leakage_risk`：背景泄露风险（重点防“看背景反推位置”）

### Step 5 · `verify_inputs.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/qc_reports/<question_id>.json` |
| **关键字段** | `qc_status`（pass / fail / needs_manual_review）/ `checks[]`（name + answer + confidence）/ `risk_flags` |
| **下游消费者** | Step 6（仅对 pass 样本定稿 prompt）、Step 7（导出过滤）、人工复核 |
| **工程用途** | 用结构化 VQA 检查输入图是否满足题目计划；不通过则重写或丢弃，最多 3 次重试 |
| **论文用途** | 论证数据集质量；可报告“通过率 / 失败模式分布” |

**字段释义**：

- `qc_status`：三状态门禄：
  - `pass`：进入 prompt 终稿
  - `fail`：丢弃或重试（最多 3 次）
  - `needs_manual_review`：转人工复核
- `checks[]`：结构化 VQA 项目（如“是否只含主体 s1”“是否背景为纯肤色”）。每项含 name / answer / confidence。
- `risk_flags`：阶段性风险旗标（如“multi-subject crop 重叠”）。汇总后进入 dataset_card。

### Step 6 · `finalize_prompts.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/prompts/final_prompts.jsonl` |
| **关键字段** | `i2v_prompt`（8–25 词英文）/ 多图 prompt 中的 reference 角色指代（image 1 / reference jacket）/ 禁词检查结果 |
| **下游消费者** | Step 7（数据集导出）、Phase 3（模型输入） |
| **工程用途** | 给模型看的最终自然语言指令；与 evaluator metadata 解耦 |
| **论文用途** | 附录中的样例题；展示 prompt 风格贴近真实用户表达 |

**字段释义**：

- `i2v_prompt`：给模型看的最终英文指令。长度 8–25 词，贴近真实用户风格。
- 多图 prompt 中的参考角色指代：统一指代方式（如 *image 1*, *the reference jacket*），使模型能对齐多图输入。
- `禁词检查结果`：是否命中 question_plan 的 forbidden_words。命中则重写，避免维度泄漏。

### Step 7 · `export_dataset.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/samples/{dimension}.jsonl` + `phase3_manifest.jsonl` + `dataset_card.md` |
| **关键字段（BenchmarkSample）** | `question_id` / `dimension` / `input_mode` / `subtype` / `difficulty` / `semantic_rarity` / `contrastive_pair_id` / `input_images[]` / `i2v_prompt` / `metadata`（target_subjects / reference_subjects / **expected_change** / **preserve_constraints** / **evaluator_hints**）/ `source_trace` / `qc` |
| **下游消费者** | Phase 3 全流程 |
| **工程用途** | Phase 2 的最终交付物；`phase3_manifest.jsonl` 是 Phase 3 唯一入口 |
| **论文用途** | 数据集发布物；reviewer 可直接根据 source_trace 验证每条样本的 provenance |

**字段释义**：

- `question_id`：题目唯一 ID。Phase 3 调度与人工标注的主键。
- `dimension` / `input_mode` / `subtype` / `difficulty` / `semantic_rarity`：题目属性五元组，Phase 3 聚合的分组键。
- `contrastive_pair_id`：如有反向题，此字段与其配对。Phase 3 可计算 contrastive consistency。
- `input_images[]`：输入图清单（含 path + role + source_type）。
- `i2v_prompt`：给模型的最终指令。
- `metadata`：**Phase 3 评测的说明书**，包含：
  - `target_subjects`：谁要变
  - `reference_subjects`：谁是参照
  - `expected_change`：要发生的变化（operation + final_state）
  - `preserve_constraints`：要保持什么
  - `evaluator_hints`：评测器提示（推荐工具、阈值、决策依据）
- `source_trace`：可追溯链：recipe_id → sample_id → ref_id。reviewer 验证用。
- `qc`：本题的 QC 结果摘要（status + risk_flags）。

### Step 8 · `audit_phase2.py`

| 项 | 说明 |
|---|---|
| **产物** | `benchmark_dataset/dataset_card.md` + 审计统计 |
| **关键字段** | 各维度 / input_mode / subtype 实际产出数；contrastive pair 覆盖率；reference quality 分布 |
| **下游消费者** | 项目 owner、Phase 3 启动前 gating |
| **工程用途** | 校验 phase3_manifest 与 samples 一致；统计 multi_image 样本质量是否达标 |
| **论文用途** | 数据集 card；论文表 1（dataset statistics）|

**字段释义**：

- `各维度实际产出数`：与 quota_plan 对照，偷减的桶不允许静默上线。
- `contrastive pair 覆盖率`：要求覆盖所有指定需要成对的桶。

## 四、Phase 3：模型评测

> Phase 3 消费 `phase3_manifest.jsonl`，对待评 I2V 模型生成视频并打分，输出 `runs/eval/{model_name}/`。

### Step 1 · `generate_videos.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/generations/<question_id>.mp4` + 每条 `GeneratedVideo` 元数据 |
| **关键字段** | `status`（success / failed / unsupported_input_mode）/ `generation_params`（seed / num_frames / fps / resolution）/ `errors` |
| **下游消费者** | Step 2（预处理）、Step 5（评测器）、报告 |
| **工程用途** | 调用待评模型；失败的样本要记录但不能中断全局评测 |
| **论文用途** | 模型对比的原始素材；`unsupported_input_mode` 计数可揭示哪些模型不支持多图 |

**字段释义**：

- `status`：三状态：
  - `success`：生成成功
  - `failed`：生成失败（超时 / OOM / API 错误）
  - `unsupported_input_mode`：模型不支持多图输入。该状态在论文中作为能力诊断依据，不计为模型质量问题。
- `generation_params`：生成参数快照（seed / num_frames / fps / resolution）。保证可复现。
- `errors`：失败原因。使质量问题可以区分于能力问题。

### Step 2 · `preprocess_videos.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/normalized_videos/` + 抽帧目录 |
| **关键字段** | `normalized_video_path` / `frames_dir` / `num_frames` / `fps` / `resolution` |
| **下游消费者** | Step 3 / 4（grounding & tracking）、所有 evaluator |
| **工程用途** | 把不同模型的输出统一到固定 fps / 分辨率；保证后续评测器可比 |
| **论文用途** | 实验设置章节中对"评测时帧率统一"的实施依据 |

**字段释义**：

- `normalized_video_path`：统一到固定 fps / 分辨率后的视频路径。所有 evaluator 只读这个。
- `frames_dir`：抽帧后的帧目录（供 grounding / tracking / VLM 不重复读取）。
- `num_frames` / `fps` / `resolution`：统一后的元信息，保证不同模型间评测可比。

### Step 3 · `grounding.py`

| 项 | 说明 |
|---|---|
| **产物** | 每帧的 bbox / mask + grounding confidence |
| **关键字段** | `target_subject_id` / `bbox_per_frame[]` / `mask_per_frame[]` / `grounding_confidence` / `tool_status` |
| **下游消费者** | Step 4（tracking）、所有 evaluator |
| **工程用途** | 把 metadata.target_subjects 在视频中真正"找到"；失败时标记 `tool_uncertain`，不简单给 E=0 |
| **论文用途** | 工具置信度报告；解释 benchmark 的可信度边界 |

**字段释义**：

- `target_subject_id`：被定位的主体在 metadata 中的 ID。来自 Phase 2 的 target_subjects。
- `bbox_per_frame[]` / `mask_per_frame[]`：逐帧的几何位置。Spatial / Motion evaluator 的原材料。
- `grounding_confidence`：逐帧检测置信度。低于阈值该帧不计入评测。
- `tool_status`：`valid` / `low_confidence` / `tool_uncertain` / `invalid_input`。**不能简单将 tool_uncertain 设为 E=0。**

### Step 4 · `tracking.py`

| 项 | 说明 |
|---|---|
| **产物** | 主体轨迹 / mask tube + tracking confidence |
| **关键字段** | `target_tube` / `displacement_curve` / `tracking_confidence` / `lost_ratio` |
| **下游消费者** | Motion / Spatial / Action / Interaction evaluator |
| **工程用途** | 给位移类与动作类维度提供时序证据；丢失比例进入 tool_confidence |
| **论文用途** | 失败案例诊断；展示 tracking 在生成视频上的有效边界 |

**字段释义**：

- `target_tube`：主体跨帧的 mask tube（三维轨迹）。作为 Motion / Action evaluator 的输入。
- `displacement_curve`：位移曲线（时间 × 位置）。Motion 维度的方向 / 速度判定依据。
- `tracking_confidence`：跨帧 tracking 置信度。进入 tool_confidence 聚合。
- `lost_ratio`：被跟丢的帧占比。过高则样本转 `tool_uncertain`。

### Step 5 · `evaluators/{dimension}.py`

| 项 | 说明 |
|---|---|
| **产物** | 每条样本的 `EvalResult`（暂存内存或临时 JSONL） |
| **关键字段** | `E` / `P` / `C` / `S` / `aggregation=exec_gated` / `lambda=0.6` / `tool_status` / `tool_confidence` / `failure_modes[]` / `auxiliary` |
| **下游消费者** | Step 6（统一打分写盘）、Step 8（聚合）、Step 9（人类对比） |
| **工程用途** | 七维度各自的核心评测逻辑；E/P/C 必须独立保存 |
| **论文用途** | 雷达图、E/P/C 分解柱状图、失败模式案例图 |

**字段释义**：

- `E`（Execution）：prompt 要求的变化是否被执行。默认 \[0,1]。
- `P`（Preservation）：保持约束是否被遵守。默认 \[0,1]。
- `C`（Coherence）：过程是否平滑、无闪烁 / 跳变 / identity drift。默认 \[0,1]。
- `S`：最终样本分数（由 Step 6 计算并写入）。
- `aggregation=exec_gated` / `lambda=0.6`：记录评分公式与权重，保证可复现。
- `tool_status` / `tool_confidence`：工具状态 + 置信度。`tool_uncertain` 不进入主表聚合。
- `failure_modes[]`：本题命中的失败模式标签集（如 `static_copy_like` / `wrong_direction`）。论文失败案例图的分类依据。
- `auxiliary`：辅助字段（如原始 iou 值 / 位移距离），供后续分析。

### Step 6 · `scoring.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/per_sample_scores.jsonl` |
| **关键字段** | 应用 `S = E * (0.6P + 0.4C)` 后的最终样本分；`tool_uncertain` 时 `S=null` |
| **下游消费者** | Step 8（聚合）、Step 9（人类一致性） |
| **工程用途** | 统一执行门控公式；防止"静态复制 + 高 P/C"刷分 |
| **论文用途** | 主表分数列；门控公式有效性的实验依据 |

**字段释义**：

- 门控公式：`S = E * (0.6 * P + 0.4 * C)`。乘法设计使“静态复制”这类退化解拿不到高分。
- `tool_uncertain` 样本：`S = null`，不进入聚合，但记录在 tool_uncertainty_report。
- `S` 范围：\[0,1]，与 E/P/C 同量纲。

### Step 7 · `baselines.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/baseline_<name>/per_sample_scores.jsonl` |
| **关键字段** | static_copy / random_motion / global_filter / camera_pan_cheat 四类 baseline 的 E/P/C/S |
| **下游消费者** | Step 8（聚合到对照表）、论文 anti-shortcut 分析 |
| **工程用途** | 验证评测器不会奖励退化解；Static Copy 总分应 < 0.10 |
| **论文用途** | 评测器可信度的关键证据章节；reviewer 最关心的 anti-shortcut 表 |

**字段释义**：

- `static_copy`：把首帧复制为静态视频。**预期总分 < 0.10**，否则说明门控失效。
- `random_motion`：随机平移 / 随机背景变化。验证模型是否能区分“乱动”与“按指令动”。
- `global_filter`：整图调色 / 滤镜。**不能在 background_dynamics 以外拿高分**。
- `camera_pan_cheat`：镜头平移伪装主体在动。**不能在 motion / spatial 拿高分**。

### Step 8 · `aggregate.py`

| 项 | 说明 |
|---|---|
| **产物** | `dimension_scores.json` + `input_mode_scores.json` + `overall_report.json` + `failure_diagnostics.json` + `tool_uncertainty_report.json` |
| **关键字段** | 按 dimension / input_mode / subtype / difficulty / semantic_rarity / source_type / contrastive_pair 七种分组聚合的 S / E / P / C；`tool_uncertain_rate` |
| **下游消费者** | Step 10（报告）、论文图表 |
| **工程用途** | 把样本级分数聚合到论文表所需粒度；同时提供 single-only / multi-only 视角 |
| **论文用途** | 主表（每个维度的 S/E/P/C）；多图 vs 单图对比图；难度桶差异图 |

**字段释义**：

- `dimension_scores.json`：七维度的 S / E / P / C 聚合。论文主表。
- `input_mode_scores.json`：`S_dim_single` 与 `S_dim_multi` 独立报告，避免不支持多图的模型被显著低估。
- `overall_report.json`：总表。含混合 / single-only / multi-only 三种口径。
- `failure_diagnostics.json`：按 failure_modes 聚合。`invalid_input` 样本仅出现在该报告中，不计入 mean。
- `tool_uncertainty_report.json`：工具不确定样本的占比与分布。论文可信度部分的补充表。
- 聚合分组键：dimension / input_mode / subtype / difficulty / semantic_rarity / source_type / contrastive_pair 七种。

### Step 9 · `human_validation.py`

| 项 | 说明 |
|---|---|
| **产物** | `human_subset_manifest.jsonl` + `human_annotations_template.csv` + `human_correlation_report.json` |
| **关键字段** | 抽样的 question_id 子集；标注表（human_E / human_P / human_C / failure_modes / artifact_severity）；Spearman / Kendall 相关性 |
| **下游消费者** | 论文人类一致性章节、Interaction 维度（默认更高比例） |
| **工程用途** | 验证自动评测器与人类判断的一致性；按 artifact severity 分桶可揭示工具失效边界 |
| **论文用途** | benchmark 可信度的关键章节；通常是 reviewer 最关心的“自动指标是否对得上人评” |

**字段释义**：

- `human_subset_manifest`：抽样题集。正式版每维度 50–100 题；pilot 每维度全量 20 题。
- `human_annotations_template.csv`：给人工标注员填写的表（human_E / human_P / human_C / failure_modes / artifact_severity）。
- `human_correlation_report`：Spearman / Kendall 相关性 + 标注者间一致性。
- 分桶报告：按 artifact severity 分桶，揭示工具在生成视频上的有效边界。
- Interaction 维度：默认人工抽样比例高于其它维度（其评测器置信度较低）。

### Step 10 · `report.py`

| 项 | 说明 |
|---|---|
| **产物** | `runs/eval/{model}/visualizations/` + 汇总 markdown / pdf |
| **关键字段** | 七维度雷达图、E/P/C 柱状图、single vs multi 对比、与 VBench-I2V 排名差、Static Copy 表、人类一致性表、典型失败案例图 |
| **下游消费者** | 论文作图、对外发布 |
| **工程用途** | 一键生成论文所需图表；统一格式便于多模型横向对比 |
| **论文用途** | 论文主图、补充材料、leaderboard 页面 |

**字段释义**：

- `七维度雷达图`：跨维度能力谱。默认轴为 S，可切换为 E / P / C。
- `E/P/C 柱状图`：揭示模型是“执行弱”还是“保持弱”。
- `single vs multi 对比`：同一模型在两种 input_mode 下的差异。
- `与 VBench-I2V 排名差`：证明本 benchmark 与质量型 benchmark 不同步，提供互补信息。
- `Static Copy 表`：anti-shortcut 主证据。
- `人类一致性表`：按维度列 Spearman / Kendall。
- `典型失败案例图`：按 failure_mode 分组的选例，附充原始首帧 + 生成视频抽帧。

## 六、一句话速览

- **Phase 1 产出"做题用的原料和说明书"**：先验包、参考资产库、可采样 recipe、维度兼容矩阵、维度隔离约束。（Step 1–5 基础分析 + Phase 1 增强 6 步）
- **Phase 2 产出"标准化考卷"**：每道题带输入图、prompt、目标变化、保持约束和评测指引。
- **Phase 3 产出"考试结果与可信度证据"**：每条样本的 E/P/C/S、维度聚合、baseline 对照、人类一致性。

三个阶段的产物共同构成**可追溯、可复现、可审计**的 I2V 组合性评测闭环。
