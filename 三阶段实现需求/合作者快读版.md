## 一、一句话讲明白这个项目

> **给"图生视频（I2V）"模型出一套带标准答案的考卷**：给模型一张图和一句话指令，看它生成的视频有没有按指令把该变的变了、不该动的没动。

---

## 二、用做菜来理解三阶段

```
Phase 1：去市场调研顾客点单 + 备好食材
Phase 2：根据调研结果设计标准化菜单
Phase 3：让不同厨师做菜，请评委按统一标准打分
```

| 阶段 | 干啥 | 类比 |
|---|---|---|
| **Phase 1** | 分析 100 万条真实 I2V 用户记录，挖出"大家常画什么"，并把首帧切成可复用素材 | 调研 + 备料 |
| **Phase 2** | 按比例出 1400 道题，每道题给好首帧、参考图、prompt 和"标准答案" | 定菜单 |
| **Phase 3** | 让每个待评模型生成视频，自动评委按 7 个维度打分 | 比赛打分 |

## 三、我们到底在测模型什么？（七个维度）

每个维度只问一件事：

| 维度 | 一句话考点 | 真人例子 |
|---|---|---|
| **属性** Attribute | 指定主体的颜色 / 状态变没变？ | 让白狗变黑，棕狗别动 |
| **动作** Action | 指定的人有没有做指定动作？ | 让左边的人挥手，右边的别挥 |
| **运动** Motion | 主体有没有按指定方向移动？ | 红球向右移到蓝盒右边 |
| **空间** Spatial | 多个物体有没有摆对位置？ | 猫在沙发上 + 绿植在沙发旁 |
| **背景** Background | 背景有没有按指定方式变？ | 天空变阴，但人物保持不变 |
| **视角** View | 镜头运动对不对？ | 镜头慢慢拉近，画面别被改写 |
| **交互** Interaction | 多个物体之间的互动对不对？ | 球撞倒积木，且先碰到再倒 |

**两条最容易搞错的边界**：

1. **动了 vs 没动**
   - 物体位置发生位移 → **运动**
   - 物体没动，只是要求最终摆对了 → **空间**
   - 例子："红球移到蓝盒右边" 是运动；"猫坐在沙发上" 是空间。

2. **一个人 vs 多个人/物**
   - 一个主体自己做动作 → **动作**
   - 多个主体之间有因果/接触/传递 → **交互**
   - 例子："女人挥手" 是动作；"女人把书递给男人" 是交互。

## 四、三阶段在做什么（白话版）

### 🥬 Phase 1：先验分析 + 食材准备

**输入**：原始 TIP-I2V 数据集（用户 prompt + 首帧）。

**核心动作**：

1. 清洗用户 prompt（去掉平台参数）。
2. 用 VLM 把首帧"看懂"：里面有什么主体、什么颜色、在哪、背景是什么。
3. 用 LLM 把 prompt 解析成"用户到底要让什么变"。
4. 把首帧切片：单独的人 / 单独的衣服 / 没人的场景 …… → 这就是"参考图素材库"（reference_bank）。
5. 统计真实分布：常见主体、常见动作、常见场景，写成 `prior_package.json`。
6. 把上面所有信息打包成"出题方案"（candidate_recipes）。

**最重要的产物**：

- `prior_package.json`：真实分布快照，告诉下游"什么常见、什么罕见"。
- `reference_bank/`：可复用素材库，让多图题不用凭空拼。
- `candidate_recipes.jsonl`：候选题方案，Phase 2 直接从这里采样。

**为什么要这么做**：避免出题人凭感觉拍脑袋。每道题最后都能追溯到真实用户数据。

---

### 📋 Phase 2：出题（合成 Benchmark）

**输入**：Phase 1 的候选题方案 + 素材库。

**核心动作**：

1. **定配额**：7 维度 × 200 题 = 1400 道（pilot 先做 7 × 20 = 140 道）。
2. **采样**：按比例从候选里抽。Spatial 80% 多图，View 95% 单图（按真实数据来定的）。
3. **设计题目**：每道题要有
   - 输入图（单图或多张参考图）
   - I2V prompt（给模型看的英文指令）
   - **标准答案**（给评委看的：什么该变、什么不该变、用什么工具评）
4. **质检**：用 VLM 反复检查输入图是否合格，不合格就重做。
5. **定稿**：导出最终数据集 `benchmark_dataset/`。

**最关键的设计**：每道题除了 prompt，还带一份**结构化的标准答案**（叫 metadata）。这样 Phase 3 的自动评委不用去猜 prompt 是什么意思，直接照着 metadata 评。

**例子**：题目 "The red ball moves to the right side of the blue box."

```
prompt: 给模型看的英文话
metadata 里偷偷写好:
  - 目标：让红球（s1）出现在蓝盒（s2）右边
  - 不许动的：蓝盒外观和位置、整体背景
  - 评的工具：grounding + tracking + 光流
```

---

### 🏆 Phase 3：评分

**输入**：Phase 2 的题 + 待评模型生成的视频。

**核心动作**：

1. 让模型按题目生成视频（每道题一个 mp4）。
2. 自动评委用 3 个分数评每条视频：
   - **E (Execution)**：模型有没有完成指定变化？
   - **P (Preservation)**：不该动的东西保住了吗？
   - **C (Coherence)**：变化过程平滑吗、有没有闪烁？
3. 算总分：

```
S = E × (0.6P + 0.4C)
```

**为什么用乘法不用加法**：因为 I2V 模型最常见的偷懒方式是"把首帧复制成静态视频"——这种视频画面超稳定，P 和 C 看起来都很高，但根本没执行变化。如果用加法平均，它能拿不低的分；用乘法门控后，E=0 就直接总分=0。

4. 跑几个"作弊基线"做对照：
   - **静态复制**：直接把首帧重复 → 应该接近 0 分
   - **整图加滤镜**：把整张图变色 → 不能在背景题拿高分
   - **整体平移**：镜头假装主体在动 → 不能在运动题拿高分

5. 抽一部分题人工标注，验证自动评委和人类评委是否一致。

## 五、四个核心设计决策（一定要理解）

### 1. 多图模式不是"增强题"，是主战场

很多 benchmark 把多图当作 stress test。我们不是。我们的 7 个维度里**每个都有多图样本**，且 Spatial 80% / Interaction 50% / Attribute 40% 都是多图。这是因为 I2V 模型未来一定要能"组合多个参考图"，这才是真正的组合性能力。

### 2. 多图素材必须可追溯

我们不从 Stable Diffusion 凭空拼参考图，**所有多图素材都来自真实 TIP-I2V 首帧的拆分**。每张参考图都记录了"我从哪条原始数据来"。这样论文里能站得住脚：多图任务不是人造的。

### 3. E/P/C 三分独立保存

不要只算总分。**E、P、C 必须分开报**。因为：
- 模型 A：E=0.9 但 P=0.3 → 改对了主体但污染了背景
- 模型 B：E=0.3 但 P=0.9 → 啥都没改，光保住了原图

总分可能差不多，但能力差异完全不同。

### 4. 工具失败不等于模型失败

如果 grounding/tracking 在某条视频上失败（比如视频里的主体被生成得严重变形），**不能简单给 E=0**。要标 `tool_uncertain`，单独统计或交人工复核。否则会冤枉模型。

## 六、合作者最容易问的几个问题

> **Q1：为什么不直接用 VBench-I2V？**
> VBench-I2V 主要看视频质量、流畅度、首帧一致性，不看"指令是否被执行"。我们填的是这块空白：给定指令，模型有没有照做。

> **Q1.5：与 T2V-CompBench 有什么区别？**
> T2V-CompBench 考 T2V（只输入文本），我们考 I2V（输入首帧 + 文本）——两者能力要求不同：I2V 多了"在给定首帧上执行变化"这个限制，评测重心从"能不能生成"转为"能不能按指令改变”。其次三点差异：题目全部源自真实用户分布（TIP-I2V）不拼词表；多图作为主维度不是压力测试；加了 subject_swap_inverse 反向配对题揭露模型是否真的在听指令。

> **Q2：1400 道题够吗？**
> 每个维度 200 道，common/rare = 80/20，难度 40/40/20，配上 contrastive pair（正反对照题），统计上够支撑维度间比较和模型间排名。pilot 阶段先 140 道跑通流程。

> **Q3：评委是 GPT 吗？**
> 不是。整个 evaluator 大量 **fork 自 T2V-CompBench 开源代码**（Spatial 几何判定、Motion DOT 跟踪、前后景分割流程、一致性评分公式）并增量修正了三个关键工程细节：Depth Anything 「值越大=越近」反直觉语义、Motion 必须重采样到 fps=8、帧 → 视频官方非线性映射公式。LLaVA-1.5 升级为**项目方后续选型的 MLLM**（需原生支持 16 帧视频输入 + Disentangled VQA 拆问 + A/B/C/D 评级）。底层视觉工具（GroundingDINO / SAM / DepthAnything / DOT / RAFT）为可插拔 backbone，全部在 `configs/evaluator_weights.yaml` 冻结哈希，同一 `evaluator_version` 下的分数才能跨论文比较。MLLM 只在高层语义拆问中被调用，不让它自由发挥。
>
> **Q3.5：为什么不直接用 T2V-CompBench 原版评测器？**
> 三个原因：（1）原版评 T2V（文生视频），不考「是否保住了首帧该保住的部分」这个 I2V 独有维度（我们的 P 维度）；（2）原版不考 multi_image 作为主维度；（3）原版不考 contrastive_pair（打乱 actor/patient 检查模型是否真听指令）。所以我们 fork 它的代码底实现，加这三件事。

> **Q4：我能贡献什么？**
> - 写某个维度的 evaluator（参考 [Phase3_模型评测.md](./Phase3_模型评测.md)）
> - 抽样人工标注（验证自动评委是否对得上人评）
> - 帮忙设计某个维度的题模板
> - 跑一个新模型并提交结果

> **Q5：为什么 View 维度多图占 5%？**
> View 测的是镜头控制（zoom/pan/tilt），主要靠单图就够了。多图组合后再做镜头控制会增加歧义，所以只留少量当压力测试。

> **Q6：和 T2V-CompBench 的 leaderboard 兼容吗？**
> 主分（E/P/C 三分门控）不直接兼容——我们多了 P 维度、多了多图、多了 swap_inverse。但我们额外导出一份 T2V-CompBench 格式的 CSV（最后一行 `Score: x`），覆盖 spatial / motion / action / interaction / attribute 5 个能对齐的维度。这样审稿人问「是否真的比 T2V-CompBench 更难/更好」时，可以直接丢他一份可打 leaderboard 的号。主表以我们门控 S 为准，附录贴 T2V-CompBench 兼容 CSV 供横比。
